# OpenCensus / OpenTracing 迁移工具开发路线图

> **核心目标**: 规划从 OpenCensus 和 OpenTracing 到 OpenTelemetry 的自动化迁移工具链

---

## 一、工具定位

### 1.1 为什么需要迁移工具？

手动迁移 OpenCensus/OpenTracing 代码存在以下挑战：

```text
手动迁移痛点:
├── 代码量大: 中大型项目可能有数千个 API 调用点
├── 容易遗漏: 某些边缘路径的 Span 创建可能遗漏
├── 语义差异: OpenTracing 的 log() 到 OTel 的 addEvent()/recordException() 需要判断
├── 测试更新: 单元测试中的 Mock 对象需要同步替换
├── 重复劳动: 每个项目都做相同的机械替换
└── 学习成本: 开发者需要深入理解两套 API 的差异

自动化工具价值:
├── 减少 80% 以上的机械修改工作
├── 降低人为遗漏和错误
├── 标准化迁移模式
├── 加速大规模代码库升级
└── 提供可审计的迁移报告
```

## 二、工具架构

### 2.1 迁移工具链

```text
迁移工具链架构:

┌─────────────────────────────────────────────┐
│           迁移分析引擎                        │
│  ├── 代码扫描: 识别所有 OC/OT API 调用点      │
│  ├── 影响评估: 统计调用点数量、文件分布        │
│  ├── 风险标记: 标识需要人工判断的复杂场景      │
│  └── 生成报告: JSON/HTML 格式的迁移分析报告   │
└─────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────┐
│           代码转换引擎                        │
│  ├── AST 解析: 基于语法树的精确替换            │
│  ├── 模式匹配: 常见 API 调用模式的自动转换     │
│  ├── 语义转换: 处理 log() → addEvent() 等    │
│  └── 保留注释: 保持原有代码注释和格式          │
└─────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────┐
│           验证引擎                           │
│  ├── 编译检查: 确保转换后代码可编译            │
│  ├── 测试映射: 自动更新单元测试中的 Mock      │
│  ├── 差异报告: 生成 before/after diff         │
│  └── 人工审查: 标记需要人工确认的关键变更      │
└─────────────────────────────────────────────┘
```

## 四、迁移规则库

### 4.1 OpenTracing → OTel 核心规则

| OpenTracing API | OpenTelemetry API | 复杂度 | 自动转换 |
|----------------|------------------|--------|---------|
| `tracer.buildSpan(name).start()` | `tracer.spanBuilder(name).startSpan()` | 低 | ✅ 自动 |
| `span.finish()` | `span.end()` | 低 | ✅ 自动 |
| `span.setTag(key, value)` | `span.setAttribute(key, value)` | 低 | ✅ 自动 |
| `span.log(fields)` | `span.addEvent(name)` 或 `span.recordException(e)` | 高 | ⚠️ 需判断 |
| `span.setBaggageItem(key, value)` | `Baggage.current().toBuilder().put(key, value)` | 中 | ✅ 自动 |
| `tracer.inject(spanCtx, format, carrier)` | `propagator.inject(context, carrier, setter)` | 中 | ✅ 自动 |
| `tracer.extract(format, carrier)` | `propagator.extract(context, carrier, getter)` | 中 | ✅ 自动 |
| `References.CHILD_OF` | `setParent()` | 低 | ✅ 自动 |
| `References.FOLLOWS_FROM` | `addLink()` | 低 | ✅ 自动 |

### 4.2 需要人工判断的场景

| 场景 | 原因 | 工具行为 |
|------|------|---------|
| `span.log({"event": "error", ...})` | 需判断是 addEvent 还是 recordException | 标记待人工确认 |
| 自定义 Carrier/Format | 需适配 TextMapPropagator 接口 | 生成模板代码 |
| 多线程 Span 传递 | OpenTracing 和 OTel 的上下文模型不同 | 标记并给出建议 |
| Metrics（OpenCensus）| Measure/View 模型差异大，需重新设计 | 生成分析报告 |

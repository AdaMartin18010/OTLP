# 范围专项批判性评价

> **评价范围**: 本项目三条内容范围（OTLP 协议、Metrics 数据模型、Logs 数据模型）
> **联动文档**: [00_范围-权威对齐矩阵](../00_范围-权威对齐矩阵.md)、[项目全面批判性评价](./02_项目_项目全面批判性评价.md)
> **最后更新**: 2025-02-10

---

## 1. 评价说明

本报告仅针对以下三条范围做专项批判性评价，输出与权威规范的对齐度、文档完整性/可操作性、缺口与风险，以及**明确意见与建议**（应做/不应做、优先级），并与 [00_范围-权威对齐矩阵](../00_范围-权威对齐矩阵.md) 联动更新。

| 范围 | 定义要点 |
|------|----------|
| **OpenTelemetry OTLP** | trace/metric/log 传输、序列化（gRPC/HTTP）、语义约定 |
| **OTLP Metrics Data Model** | pre-aggregated 时序、与 Prometheus/StatsD 双向映射 |
| **OTLP Logs Data Model** | 统一日志结构（Event、属性、资源）、多种日志格式无歧义映射 |

## 2. OTLP 协议 专项评价

### 2.1 与权威规范/实践的对齐度

- **权威来源**: opentelemetry-proto、OTLP Spec、semantic-conventions（版本见 [version-tracking](../../version-tracking/current-versions.yml)）。
- **对齐度**: 协议与编码、语义约定文档与 proto/semconv 版本一致，传输（gRPC/HTTP）与编码（Protobuf/JSON）覆盖完整。
- **版本/日期**: 文档中已引用 v1.3.x 协议与 v1.29.0 语义约定；建议在协议概述等处统一标注“与 version-tracking 同步”。

### 2.2 本项目文档/示例的完整性、准确性、可操作性

- **完整性**: 高。01_OTLP核心协议、02_Semantic_Conventions 覆盖协议与语义；协议概述已增加三条范围及文档入口。
- **准确性**: 与官方规范一致；形式化表述为项目特色。
- **可操作性**: 良好；有配置示例与多语言说明，部分高级场景可再补充 Collector 端到端示例。

### 2.3 缺口与风险

- **缺口**: 无重大缺口；三条范围入口已显式列出。
- **风险**: 若 version-tracking 未及时跟进 proto/semconv 发布，文档可能滞后。

### 2.4 意见与建议

| 建议 | 优先级 | 说明 |
|------|--------|------|
| 保持协议概述中“三条范围及文档入口”与 00_范围-权威对齐矩阵 同步更新 | P1 | 每次矩阵更新时检查协议概述链接 |
| 继续依赖 version-tracking 与季度评审，确保协议/语义版本不滞后 | P1 | 制度层面，见 [README 持续改进流程](../README.md) |
| 不必在协议层扩展非 OTLP 的私有协议说明 | 不应做 | 保持范围聚焦 |

## 3. OTLP Metrics 数据模型 专项评价

### 3.1 与权威规范/实践的对齐度

- **权威来源**: OpenTelemetry Metrics 规范、Prometheus/StatsD 与 OTLP 的映射（Collector 与社区文档）。
- **对齐度**: 数据模型与 Metrics 规范一致；pre-aggregation 与 Prometheus/StatsD 双向映射已在本项目中通过 [03_Pre-aggregation与Prometheus_StatsD映射](../../03_数据模型/02_Metrics数据模型/03_Pre-aggregation与Prometheus_StatsD映射.md) 显式覆盖。
- **版本/日期**: 与 proto 一致；Prometheus/StatsD 映射依赖 Collector 文档，建议在矩阵中注明“随 Collector 更新”。

### 3.2 本项目文档/示例的完整性、准确性、可操作性

- **完整性**: 高。01_Metrics概述、02_Metrics子类型详解、03_Pre-aggregation与Prometheus_StatsD映射 覆盖类型、聚合临时性、pre-aggregation、双向映射。
- **准确性**: 类型与聚合语义与规范一致；映射表与官方/Collector 文档对齐。
- **可操作性**: 良好；可再增加 1～2 个 Collector 配置片段（Prometheus receiver + OTLP exporter、StatsD receiver）以利落地。

### 3.3 缺口与风险

- **缺口**: 已补全 pre-aggregation 与 Prometheus/StatsD 映射；可选缺口为 Collector 实战配置示例。
- **风险**: Prometheus/StatsD 版本或语义若变更，需定期核对 Collector 文档并更新映射说明。

### 3.4 意见与建议

| 建议 | 优先级 | 说明 |
|------|--------|------|
| 保持 03_Pre-aggregation与Prometheus_StatsD映射 与 Collector 官方文档的链接与类型表一致 | P1 | 季度评审时检查 |
| 可选：在 04_核心组件 或 Metrics 目录下增加“Prometheus/StatsD 与 OTLP 桥接”的 Collector 配置示例 | P2 | 提升可操作性 |
| 不在此范围引入非 OTLP 的私有指标协议 | 不应做 | 保持范围聚焦 |

## 4. OTLP Logs 数据模型 专项评价

### 4.1 与权威规范/实践的对齐度

- **权威来源**: OpenTelemetry Logs 规范（LogRecord、Resource、Scope、Body、Severity、Trace 关联）。
- **对齐度**: 统一结构（Event/属性/Resource）与多格式映射已通过 [03_统一结构与多格式映射](../../03_数据模型/03_Logs数据模型/03_统一结构与多格式映射.md) 显式覆盖；与规范一致。
- **版本/日期**: 与 proto 一致。

### 4.2 本项目文档/示例的完整性、准确性、可操作性

- **完整性**: 高。01_Logs概述、02_Logs字段与严重性详解、03_统一结构与多格式映射 覆盖结构、字段、JSON/syslog/云格式映射。
- **准确性**: 字段与层级与规范一致；映射规则明确、无歧义。
- **可操作性**: 良好；多语言示例已有，可再增加“从某格式到 OTLP”的转换代码片段（如单文件示例）。

### 4.3 缺口与风险

- **缺口**: 已补全统一结构与多格式映射；可选缺口为更多云格式（如 GCP、Azure 日志格式）的映射行或示例。
- **风险**: 云厂商日志格式可能迭代，建议在矩阵中注明“以官方最新为准”并定期核对。

### 4.4 意见与建议

| 建议 | 优先级 | 说明 |
|------|--------|------|
| 保持 03_统一结构与多格式映射 中映射表与 OpenTelemetry Logs 规范及主流格式文档一致 | P1 | 季度评审时检查 |
| 可选：补充 GCP/Azure 等云日志格式到 LogRecord 的映射行或引用 | P2 | 扩展无歧义映射覆盖 |
| 不在此范围引入非 OTLP 的私有日志协议 | 不应做 | 保持范围聚焦 |

# 中文术语表贡献指南

> **核心目标**: 规范 OTLP 项目中技术术语的中文译法，并建立向 OpenTelemetry 国际社区贡献中文本地化成果的流程

---

## 一、术语标准化的价值

### 1.1 为什么需要标准化译法？

```text
当前中文技术社区的术语混乱现象:
├── Trace → "追踪"、"链路"、"痕迹"、"跟踪"
├── Span → "跨度"、"片段"、"区间"、"范围"
├── Baggage → "行李"、" baggage"、"上下文 baggage"
├── Exporter → "导出器"、"导出程序"、"输出器"
├── Sampler → "采样器"、"取样器"、"抽样器"
└── Instrumentation → "插桩"、"埋点"、"仪表化"、"检测"

标准化价值:
├── 降低学习门槛: 同一术语全书一致
├── 便于检索: 读者可通过统一关键词查找
├── 社区对齐: 与云原生社区通用译法保持一致
├── 国际化桥梁: 为官方中文文档提供参考
└── 学术严谨: 论文和出版物使用标准译法
```

## 二、术语译法原则

### 2.1 翻译策略矩阵

| 策略 | 适用场景 | 示例 |
|------|---------|------|
| **意译优先** | 概念有成熟中文对应 | Trace → 追踪, Metric → 指标 |
| **音译+意译** | 专有名词或品牌 | OpenTelemetry → OpenTelemetry（保留英文） |
| **造词** | 全新概念无对应 | Baggage → 行李上下文 |
| **保留英文** | 技术缩写或无需翻译 | OTLP, SDK, API, gRPC |
| **混合使用** | 首次出现时双语标注 | 跨度（Span） |

### 2.2 优先级规则

```text
译法选择优先级:

1. 官方中文文档已有译法 → 优先采用
   └── 参考: Kubernetes 官方中文文档、CNCF  glossary

2. 云原生社区广泛接受的译法 → 次优先采用
   └── 参考: 阿里云、腾讯云技术文档

3. 学术界常用译法 → 第三优先
   └── 参考: 计算机科学教材、IEEE 中文出版物

4. 本项目独创译法 → 最后选择，需充分论证
   └── 要求: 在术语表中记录创制理由
```

### 2.3 禁用做法

| 做法 | 问题 | 正确替代 |
|------|------|---------|
| 同一文档中混用多个译法 | 读者困惑 | 全文统一一个译法 |
| 过度翻译（所有词都译） | 技术文档读起来像文学翻译 | 缩写和专业名词保留英文 |
| 使用生僻字或方言 | 增加理解成本 | 使用通用词汇 |
| 逐字直译不顾语境 | 语义偏差 | 结合技术语境意译 |

## 三、术语表格式规范

### 3.1 标准术语表条目

每个术语条目应包含以下字段：

```markdown
| 英文术语 | 中文译法 | 状态 | 说明 |
|---------|---------|------|------|
| Span | 跨度 | 稳定 | OTel 官方推荐，社区广泛使用 |
```

字段说明:

- **英文术语**: 官方英文原词（大小写与官方一致）
- **中文译法**: 标准中文翻译
- **状态**: 稳定 / 候选 / 实验 / 废弃
  - 稳定: 已在本项目全量使用，社区接受
  - 候选: 提议中，等待社区反馈
  - 实验: 新创译法，试用观察
  - 废弃: 不再使用，记录历史
- **说明**: 译法来源、理由、替代方案

### 3.2 术语表文件位置

```text
术语表文件:
knowledge/00_知识中心/05_术语表/
├── 01_中英对照表.md          # 核心术语对照
```

## 四、贡献流程

### 4.1 新增术语

```text
1. 检查现有术语表
   └── 确认该术语是否已存在

2. 研究译法
   ├── 检索官方中文文档
   ├── 检索主流云厂商文档
   ├── 检索学术文献
   └── 在社区中发起讨论（如需要）

3. 提交提案
   ├── 格式: GitHub Issue 或 Pull Request
   ├── 内容:
   │   ├── 英文术语
   │   ├── 提议中文译法
   │   ├── 候选译法（如有竞争方案）
   │   ├── 理由和依据
   │   └── 使用示例
   └── 标签: terminology

4. 社区评审
   ├── 至少 2 名维护者审阅
   ├── 关注: 一致性、准确性、可理解性
   └── 周期: 通常 1-2 周

5. 合并与同步
   ├── 合并到术语表
   ├── 更新所有引用该术语的文档
   └── 记录变更日志
```

### 4.2 修改现有术语

修改已稳定的术语需要更高门槛：

```text
修改条件（满足其一即可）:
├── 官方规范发布了新的推荐译法
├── 发现现有译法存在严重歧义或误导
├── 云原生社区广泛采用了新的译法
└── 术语本身的技术含义发生了重大变化

修改流程:
├── 提交 Issue 说明修改理由
├── 评估影响范围（涉及多少文档）
├── 维护者讨论并投票决定
├── 如通过，批量更新所有相关文档
└── 在变更日志中记录术语迁移
```

## 五、向国际社区贡献

### 5.1 OpenTelemetry 中文本地化

OpenTelemetry 官方正在推进多语言文档。本项目可以向社区贡献：

| 贡献类型 | 目标 | 流程 |
|---------|------|------|
| 术语建议 | OpenTelemetry 官方中文术语表 | 在 opentelemetry.io 仓库提交 Issue |
| 文档翻译 | 官方文档中文版 | 加入 OpenTelemetry Docs SIG |
| 规范注释 | 规范中的术语解释 | 在 opentelemetry-specification 仓库提交 PR |

### 5.2 贡献检查清单

- [ ] 术语译法已通过本项目内部评审
- [ ] 译法在至少 3 篇文档中试用过
- [ ] 提供了英文术语的上下文使用示例
- [ ] 解释了为何选择此译法而非其他方案
- [ ] 如果创制了新词，解释了构词逻辑

# RFC 2119 关键字合规规范

> **标准来源**: IETF RFC 2119 / RFC 8174
> **应用范围**: OTLP 项目所有技术文档、规范解析、最佳实践文档
> **核心目标**: 与 OpenTelemetry 官方规范的合规性关键字体系对齐，提升文档的精确性和可执行性

---

## 一、背景

OpenTelemetry 官方规范在所有技术文档中使用 IETF RFC 2119 / RFC 8174 定义的关键字来表达要求的严格程度：

> "The keywords 'MUST', 'MUST NOT', 'REQUIRED', 'SHALL', 'SHALL NOT', 'SHOULD', 'SHOULD NOT', 'RECOMMENDED', 'NOT RECOMMENDED', 'MAY', and 'OPTIONAL' in this document are to be interpreted as described in BCP 14 [RFC2119] [RFC8174] when, and only when, they appear in all capitals, as shown here."

这些关键字不仅是英语语气词，而是具有**严格技术法律含义**的合规标记。

---

## 二、关键字定义与中文译法

### 2.1 强制性要求

| 英文关键字 | 中文译法 | 技术含义 | 合规后果 |
|-----------|---------|---------|---------|
| **MUST** | **必须** | 定义规范的绝对要求 | 不满足即视为**不合规** |
| **MUST NOT** | **禁止 / 不得** | 定义绝对的禁止 | 违反即视为**不合规** |
| **REQUIRED** | **必需的** | 与 MUST 同义 | 不满足即视为**不合规** |
| **SHALL** | **应当** | 与 MUST 同义（RFC 2119 原义）| 不满足即视为**不合规** |
| **SHALL NOT** | **不应当** | 与 MUST NOT 同义 | 违反即视为**不合规** |

### 2.2 建议性要求

| 英文关键字 | 中文译法 | 技术含义 | 合规后果 |
|-----------|---------|---------|---------|
| **SHOULD** | **应该 / 推荐** | 在特定情况下可能有正当理由忽略，但需理解全部含义并谨慎权衡 | 可以选择不遵循，但**必须有合理理由** |
| **SHOULD NOT** | **不推荐 / 不建议** | 在特定情况下可能有正当理由的行为，但需理解全部含义 | 可以选择这样做，但**必须有合理理由** |
| **RECOMMENDED** | **推荐的** | 与 SHOULD 同义 | 同上 |
| **NOT RECOMMENDED** | **不推荐的** | 与 SHOULD NOT 同义 | 同上 |

### 2.3 可选性要求

| 英文关键字 | 中文译法 | 技术含义 | 合规后果 |
|-----------|---------|---------|---------|
| **MAY** | **可以 / 允许** | 定义规范中真正可选的要素 | 实现者可自行决定，不影响合规性 |
| **OPTIONAL** | **可选的** | 与 MAY 同义 | 同上 |

## 三、在 OTLP 项目文档中的应用规范

### 3.1 使用范围

以下类型的文档**应当**引入 RFC 2119 关键字标记：

- [ ] 协议规范解析文档（OTLP 协议、gRPC/HTTP 传输层）
- [ ] SDK 实现指南（配置、采样、导出器接口）
- [ ] 语义约定文档（属性命名、值类型、必需/可选标记）
- [ ] 安全实践文档（TLS 配置、认证要求）
- [ ] 性能规范文档（延迟要求、资源限制）
- [ ] Collector 配置规范（处理器行为、管道约束）

以下类型的文档**可以**不严格使用，但**推荐**参考：

- [ ] 行业案例与实践总结
- [ ] 概念介绍与入门教程
- [ ] 趋势分析与技术展望
- [ ] 故障排查指南（但涉及修复步骤时应使用）

### 3.2 标记格式

在 Markdown 文档中，关键字**必须**使用以下格式：

```markdown
<!-- 方式一：全大写英文 + 中文注释（推荐用于规范文档） -->
客户端 **MUST（必须）** 在请求头中设置 `Content-Type`。

<!-- 方式二：纯中文（推荐用于面向中文用户的实践文档） -->
客户端**必须**在请求头中设置 `Content-Type`。

<!-- 方式三：表格中的等级标记 -->
| 要求等级 | 说明 |
|---------|------|
| 必须 (MUST) | 不满足即不合规 |
| 推荐 (SHOULD) | 不满足需有合理理由 |
| 可选 (MAY) | 自行决定 |
```

### 3.3 与现有文档的整合

对于已存在的文档，按以下优先级逐步引入：

| 优先级 | 文档类别 | 引入方式 |
|--------|---------|---------|
| P0 | 核心协议层所有文档 | 全面审校，在关键约束句中加入关键字 |
| P1 | 核心实现层 SDK/Collector 文档 | 在接口约定和行为约束处加入关键字 |
| P2 | 部署运维层文档 | 在安全要求和配置约束处加入关键字 |
| P3 | 其他层级文档 | 视内容性质决定是否引入 |

## 四、典型应用场景示例

### 4.1 协议规范示例

```markdown
### OTLP/gRPC 传输要求

- 服务端 **MUST（必须）** 支持 `none` 和 `gzip` 两种压缩选项。
- 客户端 **SHOULD（应该）** 实现并发 Unary 调用以提升吞吐量。
- 客户端 **MUST NOT（不得）** 在收到 `partial_success` 响应后重试请求。
- 服务端 **MAY（可以）** 使用 `ResourceExhausted` 状态码表示背压信号。
```

### 4.2 SDK 实现示例

```markdown
### Batch Span Processor 规范

- `maxQueueSize` **MUST（必须）** 有默认值，且默认值 **SHOULD NOT（不应）** 超过 `2048`。
- 当队列满时，实现 **MUST（必须）** 提供可配置的行为：丢弃或阻塞。
- 默认行为 **SHOULD（应该）** 为丢弃，以防止业务线程阻塞。
- 实现 **MAY（可以）** 提供指标暴露当前队列长度和丢弃计数。
```

### 4.3 安全规范示例

```markdown
### mTLS 配置要求

- 生产环境 **MUST（必须）** 启用 TLS 加密传输。
- 涉及敏感数据的场景 **SHOULD（应该）** 启用双向 TLS（mTLS）。
- 自签名证书 **MUST NOT（禁止）** 用于生产环境，除非通过私有 CA 严格管理。
- 证书轮换机制 **MAY（可以）** 通过 OpenTelemetry Collector 的 `tls` 配置自动处理。
```

## 五、合规检查清单

在发布或更新技术文档时，检查：

- [ ] 文档中涉及行为约束的句子是否使用了规范化的关键字？
- [ ] MUST / MUST NOT 是否仅用于真正的硬性要求？
- [ ] SHOULD / SHOULD NOT 是否给出了足够的理由空间，而非被当作 MUST 使用？
- [ ] MAY / OPTIONAL 是否明确标记了可选要素，避免用户误以为必需？
- [ ] 同一文档中的关键字使用是否前后一致？
- [ ] 中文译法是否遵循本规范的定义（必须/应该/可以）？

## 六、常见错误

| 错误 | 示例 | 正确做法 |
|------|------|---------|
| 过度使用 MUST | "你 MUST 考虑性能影响" | "你 **应该** 考虑性能影响"（这是建议，不是硬性要求）|
| SHOULD 被当作 MUST | "实现 SHOULD 支持并发"（实际意思是必须支持）| 如果必须支持，写 "实现 **必须** 支持并发" |
| 使用模糊词替代 | "实现需要支持重试" | "实现 **必须** 支持指数退避重试策略" |
| 大小写不规范 | "must" / "Should" / "MAY" | 统一使用全大写 **MUST** / **SHOULD** / **MAY** |

> **总结**: RFC 2119 关键字体系是技术规范的"精确语法"。在 OTLP 项目中引入这套标记，不仅是为了与 OpenTelemetry 官方规范对齐，更是为了消除中文技术文档中常见的模糊性（如"应该""建议""最好"等口语化表达），让文档成为可执行、可验证、可审计的技术契约。

## 1. 1.0稳定版目标详解

### 1.1 配置标准化

```yaml
# OBI声明式配置示例 (2026目标)
apiVersion: opentelemetry.io/v1alpha1
kind: eBPFInstrumentation
metadata:
  name: my-app-instrumentation
spec:
  # 目标选择
  selector:
    matchLabels:
      app: my-service

  # eBPF配置
  ebpf:
    enabled: true
    sampling_rate: 100
    buffer_size: 100MB

  # 协议支持
  protocols:
    http:
      enabled: true
    grpc:
      enabled: true
    sql:
      enabled: true
    redis:
      enabled: true

  # 导出配置
  export:
    endpoint: http://otel-collector:4318
    protocol: otlp-http
    batch_size: 100

  # 资源属性
  resource:
    service.name: my-service
    service.version: "1.0.0"
```

## 2. 混合插桩模式

### 2.1 什么是混合插桩

混合插桩结合eBPF零代码插桩和SDK手动插桩的优势：

```
┌─────────────────────────────────────────────────────────────────┐
│                      混合插桩架构                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   应用代码                                                      │
│   ├── 业务代码 (手动SDK插桩)                                    │
│   │   └── 自定义span、业务属性                                  │
│   │                                                             │
│   └── 框架/库代码 (eBPF自动插桩)                                │
│       └── HTTP/gRPC/数据库调用                                  │
│                                                                 │
│   Linux内核 (eBPF)                                             │
│   ├── 自动捕获网络调用                                          │
│   ├── 自动关联context                                          │
│   └── 避免重复仪器化                                            │
│                                                                 │
│   结果: 统一、完整的可观测性                                     │
│   - eBPF提供基础设施层可见性                                    │
│   - SDK提供业务层可见性                                         │
│   - 两者协同，不重复                                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 混合插桩优势

| 场景 | 纯SDK | 纯eBPF | 混合 |
|:---|:---:|:---:|:---:|
| **快速启动** | ❌ 需要代码修改 | ✅ 零代码 | ✅ 零代码基础 + SDK增强 |
| **业务上下文** | ✅ 完整 | ❌ 有限 | ✅ 完整 |
| **基础设施覆盖** | ⚠️ 依赖库支持 | ✅ 全面 | ✅ 全面 |
| **第三方服务** | ❌ 无法观测 | ✅ 自动 | ✅ 自动 |
| **性能开销** | 中 | 低 | 低 |

### 2.3 2026年混合插桩目标

```yaml
目标功能:
  1. 智能检测:
     - 自动检测应用是否已有SDK插桩
     - 避免重复创建span
     - 避免属性冲突

  2. 上下文协同:
     - eBPF读取SDK的trace context
     - SDK读取eBPF的span信息
     - 统一的baggage传播

  3. 数据关联:
     - eBPF span作为SDK span的child
     - 共享trace ID
     - 一致的资源属性

  4. 指标统一:
     - eBPF和SDK的metrics可关联
     - 统一的exemplar支持
     - 一致的标签命名
```

## 3. 工具对比

### 3.1 eBPF可观测性工具对比 (2026)

| 工具 | 类型 | 语言支持 | 特点 | 2026状态 |
|:---|:---|:---|:---|:---:|
| **OBI** | 官方 | Go,Python,Node,Java,Ruby,.NET | 标准OTel格式，混合插桩 | 🟢 1.0稳定 |
| **Beyla** | Grafana | Go,Python,Node,Java,Rust | Grafana生态，易用 | 🟢 稳定 |
| **Odigos** | 商业 | Go,Python,Node,Java,.NET | K8s原生，分布式追踪 | 🟢 稳定 |
| **Pixie** | New Relic | Go,C/C++,Python,Node,Java | 实时查询，K8s专用 | 🟢 稳定 |

### 3.2 选择建议

```yaml
选择OBI如果:
  - 需要官方OpenTelemetry支持
  - 计划使用混合插桩模式
  - 需要多语言一致的体验
  - 希望参与社区共建

选择Beyla如果:
  - 使用Grafana栈
  - 需要简单快速上手
  - 不需要混合插桩

选择Odigos如果:
  - 纯Kubernetes环境
  - 需要最强的分布式追踪
  - 不需要修改代码

选择Pixie如果:
  - 需要实时调试能力
  - 使用New Relic
  - 可以接受K8s限制
```

## 4. 生产部署建议

### 4.1 Kubernetes部署

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: obi-instrumentation
spec:
  selector:
    matchLabels:
      app: obi
  template:
    spec:
      containers:
        - name: obi
          image: otel/obi:1.0
          securityContext:
            privileged: true  # eBPF需要
          resources:
            limits:
              memory: 512Mi
              cpu: 500m
          volumeMounts:
            - name: bpffs
              mountPath: /sys/fs/bpf
            - name: debugfs
              mountPath: /sys/kernel/debug
      volumes:
        - name: bpffs
          hostPath:
            path: /sys/fs/bpf
        - name: debugfs
          hostPath:
            path: /sys/kernel/debug
```

### 4.2 性能预期

| 指标 | 典型值 | 备注 |
|:---|:---:|:---|
| **CPU开销** | 1-5% | 取决于采样率 |
| **内存占用** | 100-300MB | OBI agent |
| **网络开销** | <1% | OTLP数据 |
| **延迟影响** | <1ms | 通常可忽略 |

## 2. Isolation Forest异常检测处理器

### 2.1 功能简介

**isolationforest processor**使用Isolation Forest算法对Metrics数据进行实时异常检测，无需预定义阈值。

```
┌─────────────────────────────────────────────────────────────────┐
│                 Isolation Forest 工作原理                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  核心思想: 异常点是"容易被隔离"的点                              │
│                                                                 │
│  正常数据点: 需要很多次随机分割才能隔离                          │
│     ●  ●    ●                                                      │
│  ●        ●  ●                                                   │
│    ●  ●      ●        ←  需要很多分割                            │
│      ●    ●  ●                                                   │
│  ●  ●  ●    ●                                                    │
│                                                                 │
│  异常数据点: 很快就被隔离                                        │
│     ●                                                            │
│  ───┼───  ←  一次分割就隔离                                       │
│     ●  ●    ●                                                      │
│                                                                 │
│  隔离路径长度 → 异常分数                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 配置示例

```yaml
processors:
  isolationforest:
    # 训练参数
    max_samples: 10000          # 训练样本数
    max_features: 3             # 特征数
    n_estimators: 100           # 树的数量
    contamination: 0.01         # 异常比例预期 (1%)

    # 检测参数
    threshold: 0.6              # 异常阈值 (0-1)

    # 特征选择
    feature_keys:
      - http_request_duration
      - cpu_usage
      - memory_usage

    # 输出配置
    anomaly_attribute: "is_anomaly"
    anomaly_score_attribute: "anomaly_score"

    # 告警配置
    alert_on_anomaly: true
    alert_threshold: 0.8
```

### 2.3 完整使用场景

```yaml
receivers:
  prometheus:
    config:
      scrape_configs:
        - job_name: 'app-metrics'
          static_configs:
            - targets: ['localhost:8080']

processors:
  # 1. 批处理
  batch:
    timeout: 10s
    send_batch_size: 1024

  # 2. 异常检测
  isolationforest:
    max_samples: 5000
    max_features: 2
    n_estimators: 50
    contamination: 0.05
    threshold: 0.7
    feature_keys:
      - http_server_duration
      - http_client_duration
    anomaly_attribute: "is_anomaly"
    anomaly_score_attribute: "anomaly_score"

  # 3. 属性添加
  resource:
    attributes:
      - key: environment
        value: production
        action: upsert

exporters:
  prometheusremotewrite:
    endpoint: https://prometheus-backend/api/v1/write

  otlp/alert:
    endpoint: alertmanager:4317
    sending_queue:
      enabled: true

service:
  pipelines:
    metrics:
      receivers: [prometheus]
      processors: [batch, isolationforest, resource]
      exporters: [prometheusremotewrite]

    # 异常指标单独路由到告警系统
    metrics/alert:
      receivers: [prometheus]
      processors: [batch, isolationforest]
      exporters: [otlp/alert]
```

|------|---------|----------|------|
| 基础配置 | 5-10% | 100MB | 10ms |

## 3. Interval指标聚合处理器

### 3.1 功能简介

**interval processor**用于按时间窗口聚合Metrics数据，支持降采样和窗口计算。

### 3.2 配置示例

```yaml
processors:
  interval:
    # 时间窗口配置
    interval: 60s               # 聚合窗口大小

    # 聚合函数
    aggregations:
      - metric_name: http_request_duration
        aggregation: p99        # 支持: sum, avg, min, max, count, p50, p90, p99

      - metric_name: cpu_usage
        aggregation: avg

      - metric_name: memory_usage
        aggregation: max        # 保留峰值

    # 输出配置
    passthrough:
      gauge: true               # 保留最新值
      cumulative: true          # 保留累计值

    # 精确时间窗口
    exact_intervals: true
```

### 3.3 使用场景

```
┌─────────────────────────────────────────────────────────────────┐
│                  Interval处理器使用场景                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  场景1: 指标降采样                                               │
│  ─────────────────                                               │
│  原始: 每秒采集 CPU使用率 (86400点/天)                          │
│  聚合: 每分钟平均 CPU使用率 (1440点/天)                         │
│  节省: 98%存储空间                                              │
│                                                                 │
│  场景2: 计算窗口统计                                             │
│  ─────────────────                                               │
│  输入: http_request_duration 原始值                            │
│  输出: http_request_duration_p99 (99分位)                      │
│                                                                 │
│  场景3: 异常平滑                                                 │
│  ─────────────────                                               │
│  原始: 带有尖刺的CPU使用率                                       │
│  聚合: 平滑后的趋势                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.4 完整配置

```yaml
processors:
  interval:
    interval: 300s              # 5分钟窗口

    # 时间戳对齐
    timestamp_alignment: start  # start, middle, end

    aggregations:
      # 延迟指标 - 计算P99
      - metric_match: ".*_duration.*"
        aggregation: p99

      # 计数指标 - 求和
      - metric_match: ".*_count"
        aggregation: sum

      # 使用率 - 平均
      - metric_match: ".*_usage"
        aggregation: avg

      # 温度/特殊 - 最大
      - metric_match: ".*_temperature"
        aggregation: max

    # 跳过某些指标
    exclude_metrics:
      - system_uptime
      - process_start_time
```

## 4. Log DeDuplication日志去重处理器

### 4.1 功能简介

**logdedup processor**检测并合并重复日志，显著减少日志存储成本和传输带宽。

### 4.2 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                  Log DeDuplication 工作原理                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  输入日志流:                                                    │
│  [10:00:01] ERROR Connection failed to database                │
│  [10:00:02] ERROR Connection failed to database     ← 重复     │
│  [10:00:03] ERROR Connection failed to database     ← 重复     │
│  [10:00:04] INFO  Connection established                      │
│  [10:00:05] ERROR Connection failed to database     ← 重复     │
│                                                                 │
│  输出日志流:                                                    │
│  [10:00:01] ERROR Connection failed to database                │
│              [重复 3次, 最后 10:00:03]                         │
│  [10:00:04] INFO  Connection established                      │
│  [10:00:05] ERROR Connection failed to database                │
│              [重复 1次, 最后 10:00:05]                         │
│                                                                 │
│  去重键: body + severity + 其他属性                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.3 配置示例

```yaml
processors:
  logdedup:
    # 去重窗口
    interval: 30s               # 30秒去重窗口

    # 去重键配置
    log_identity_attributes:
      - body                   # 日志内容
      - severity_number        # 日志级别
      - attributes["service.name"]
      - attributes["error.type"]

    # 去重行为
    emit_first_log_immediately: true    # 立即发送第一条
    emit_repeated_log_after_interval: true

    # 输出元数据
    add_count_attribute: true
    count_attribute_name: "dedup.count"

    add_first_occurrence_timestamp: true
    first_occurrence_attribute_name: "dedup.first_timestamp"

    # 去重限制
    max_log_count: 1000         # 窗口内最大日志数
    max_log_size_bytes: 1048576 # 1MB
```

### 4.4 高级配置

```yaml
processors:
  logdedup:
    interval: 60s

    # 条件去重
    include:
      match_type: regexp
      log_bodies:
        - ".*ERROR.*"
        - ".*WARN.*"

    exclude:
      match_type: strict
      log_bodies:
        - "Transaction committed"

    # 特殊处理
    emit_no_change_interval: 60s    # 即使没有变化也定期发送

    # 资源属性复制
    copy_resource_attributes: true

    # 超时控制
    log_flush_timeout: 5s
```

### 4.5 效果评估

| 场景 | 原始日志量 | 去重后 | 压缩率 |
|------|-----------|--------|--------|
| 数据库连接错误 | 10000条/小时 | 50条/小时 | 99.5% |
| 访问日志 | 100000条/小时 | 80000条/小时 | 20% |
| 系统日志 | 5000条/小时 | 3000条/小时 | 40% |

## 5. Remote Tap远程数据流处理器

### 5.1 功能简介

**remotetap processor**允许实时流式查看Collector处理的数据，用于调试和监控。

### 5.2 配置示例

```yaml
processors:
  remotetap:
    # 服务端配置
    endpoint: localhost:12001

    # 认证配置
    auth:
      authenticator: basicauth/server

    # TLS配置
    tls:
      cert_file: /etc/otel/server.crt
      key_file: /etc/otel/server.key

    # 限流配置
    limit: 100                  # 最大并发连接数

    # 采样配置
    sampling_rate: 0.01         # 1%采样 (调试时可设为1.0)

extensions:
  basicauth/server:
    htpasswd:
      file: /etc/otel/htpasswd
```

### 5.3 客户端连接

```bash
# 使用otel-cli连接remotetap
otel-cli tap --endpoint localhost:12001 \
  --processor remotetap \
  --signal traces

# 或使用websocat进行WebSocket连接
websocat wss://collector:12001/v1/tap
```

## 6. 处理器组合使用

### 6.1 推荐Pipeline架构

```yaml
processors:
  # 1. 预处理
  memory_limiter:
    check_interval: 1s
    limit_mib: 1500

  # 2. 批处理
  batch:
    timeout: 10s
    send_batch_size: 1024

  # 3. 异常检测 (Metrics)
  isolationforest:
    max_samples: 1000
    threshold: 0.7

  # 4. 指标聚合 (Metrics)
  interval:
    interval: 60s
    aggregations:
      - metric_match: ".*_duration"
        aggregation: p99

  # 5. 日志去重 (Logs)
  logdedup:
    interval: 30s
    log_identity_attributes: [body, severity_number]

  # 6. 资源增强
  resource:
    attributes:
      - key: processed.by
        value: collector-v0.147.0
        action: upsert

service:
  pipelines:
    metrics:
      receivers: [otlp]
      processors:
        - memory_limiter
        - batch
        - isolationforest
        - interval
        - resource
      exporters: [prometheusremotewrite]

    logs:
      receivers: [otlp]
      processors:
        - memory_limiter
        - batch
        - logdedup
        - resource
      exporters: [elasticsearch]
```

## 7. 性能调优建议

### 7.1 Isolation Forest调优

| 参数 | 建议值 | 说明 |
|------|--------|------|
| max_samples | 1000-10000 | 样本越多越准确，但内存越大 |
| n_estimators | 50-100 | 树越多越稳定，但CPU越高 |
| contamination | 0.01-0.05 | 根据实际异常比例调整 |
| threshold | 0.6-0.8 | 阈值越低越敏感 |

### 7.2 Interval处理器调优

| 场景 | interval | 建议 |
|------|----------|------|
| 高频指标 | 10s | 实时监控 |
| 标准监控 | 60s | 平衡精度和存储 |
| 长期趋势 | 300s | 节省存储 |

### 7.3 Log Dedup调优

| 参数 | 建议值 | 说明 |
|------|--------|------|
| interval | 10-60s | 窗口越大压缩率越高，但延迟越大 |
| max_log_count | 1000-10000 | 防止内存溢出 |

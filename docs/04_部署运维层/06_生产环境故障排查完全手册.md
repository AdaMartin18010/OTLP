## 1. 故障排查方法论

### 1.1 标准化排查流程

```
┌─────────────────────────────────────────────────────────────────┐
│                    OTLP故障排查五步法                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  第1步: 现象确认                                                  │
│    □ 收集用户报告的问题描述                                        │
│    □ 确认受影响的服务和信号类型                                    │
│    □ 记录问题发生时间范围                                          │
│    □ 检查是否有相关变更(部署/配置/基础设施)                         │
│                                                                 │
│  第2步: 范围界定                                                  │
│    □ 确定问题影响范围(单个服务/多个服务/全局)                       │
│    □ 识别数据流路径中的故障点                                       │
│    □ 区分是采集端、传输端还是存储端问题                              │
│                                                                 │
│  第3步: 数据采集                                                  │
│    □ 收集日志、指标、追踪数据                                       │
│    □ 使用zPages、health check、pprof等诊断工具                      │
│    □ 对比正常与异常期间的系统状态                                   │
│                                                                 │
│  第4步: 根因分析                                                  │
│    □ 基于收集的数据定位根本原因                                     │
│    □ 使用鱼骨图/5Why等分析方法                                      │
│    □ 验证假设并排除干扰因素                                         │
│                                                                 │
│  第5步: 解决与验证                                                │
│    □ 实施修复方案                                                  │
│    □ 验证问题解决                                                  │
│    □ 更新监控和预防措施                                            │
│    □ 撰写故障报告                                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 数据流检查清单

```yaml
# 数据流完整性检查
data_flow_checklist:
  application_layer:
    - SDK是否正确初始化
    - 采样配置是否生效
    - 资源属性是否正确附加
    - 上下文传播是否完整
  
  network_layer:
    - 网络连通性测试
    - 防火墙规则检查
    - DNS解析是否正常
    - 带宽是否充足
  
  collector_layer:
    - Receivers是否正常工作
    - Processors是否正确处理
    - Exporters是否成功发送
    - 队列状态检查
  
  storage_layer:
    - 后端存储是否可访问
    - 存储容量是否充足
    - 查询性能是否正常
    - 数据保留策略是否生效
```

## 2. 数据流故障诊断

### 2.1 数据丢失诊断树

```
数据丢失报告
     │
     ├─ 是否所有服务都受影响?
     │   ├─ 是 → 检查Collector全局配置
     │   │        ├─ 检查OTLP Receivers状态
     │   │        ├─ 检查Memory Limiter触发
     │   │        └─ 检查Exporters连接状态
     │   │
     │   └─ 否 → 仅特定服务受影响
     │            ├─ 检查SDK配置
     │            │   ├─ 采样率设置
     │            │   ├─ 批量发送配置
     │            │   └─ 资源属性过滤
     │            │
     │            └─ 检查网络连通性
     │                 ├─ 从服务到Collector
     │                 └─ Collector到后端
     │
     └─ 是否间歇性丢失?
         ├─ 是 → 检查资源竞争
         │        ├─ CPU使用率峰值
         │        ├─ 内存不足
         │        ├─ 队列满载
         │        └─ 网络波动
         │
         └─ 否 → 持续性丢失
                  ├─ 配置错误
                  ├─ 认证失败
                  └─ 后端不可用
```

## 3. Collector故障排查

### 3.1 Collector启动失败

```yaml
# 启动失败诊断流程
startup_failure_diagnosis:
  common_causes:
    - 配置文件语法错误
    - 端口占用
    - 权限不足
    - 组件缺失
    - 内存不足
  
  diagnosis_steps:
    step_1_validate_config:
      command: "otelcol-contrib validate --config=file:/etc/otel/config.yaml"
      expected_output: "Config validation passed"
    
    step_2_check_ports:
      command: "netstat -tlnp | grep -E '4317|4318|13133'"
      check: "确认端口未被占用"
    
    step_3_check_permissions:
      command: "ls -la /etc/otel/"
      check: "确认配置文件可读，证书文件可访问"
    
    step_4_check_components:
      command: "otelcol-contrib components"
      check: "确认所需组件已编译"
    
    step_5_memory_check:
      command: "free -h"
      check: "确认可用内存 > 512MB"
  
  error_patterns:
    "cannot unmarshal the configuration":
      cause: 配置文件语法错误
      solution: 使用YAML验证工具检查
    
    "listen tcp :4317: bind: address already in use":
      cause: 端口被占用
      solution: 找出占用进程或更换端口
    
    "permission denied":
      cause: 权限不足
      solution: 检查文件权限或使用sudo
```

### 3.2 OOM (内存不足) 故障

```yaml
# OOM故障诊断
oom_diagnosis:
  symptoms:
    - Collector进程被Kill
    - 日志中出现"Out of memory"
    - 系统日志显示OOM Killer活动
    - 内存使用图表突然下降
  
  root_causes:
    memory_limit_too_high:
      description: Memory Limiter限制高于实际可用内存
      check: "limit_mib > container_memory_limit"
      solution: |
        processors:
          memory_limiter:
            limit_mib: 1500  # 设置为容器限制的75%
    
    queue_size_too_large:
      description: 发送队列设置过大
      check: "queue_size * avg_data_size > available_memory"
      solution: |
        exporters:
          otlp:
            sending_queue:
              queue_size: 5000  # 减小队列大小
    
    batch_size_too_large:
      description: 批处理大小设置过大
      check: "send_batch_size * avg_item_size > memory_limit"
      solution: |
        processors:
          batch:
            send_batch_size: 1024  # 减小批大小
            send_batch_max_size: 2048
    
    memory_leak:
      description: 内存泄漏
      detection: "内存使用持续增长，不释放"
      solution: |
        1. 启用pprof收集heap profile
        2. 分析内存分配热点
        3. 升级到修复版本
  
  prevention:
    - 设置合理的Memory Limiter
    - 监控内存使用趋势
    - 设置容器内存限制
    - 配置OOM通知
```

### 3.3 队列溢出处理

```yaml
# 队列溢出诊断与处理
queue_overflow_handling:
  symptoms:
    - 日志中出现"sending queue is full"
    - otelcol_queue_size接近otelcol_queue_capacity
    - 数据导出延迟增加
    - 后端接收速率下降
  
  diagnosis_commands:
    check_queue_status: |
      curl -s http://otel-collector:8888/metrics | grep "otelcol_queue"
    
    check_exporter_rate: |
      curl -s http://otel-collector:8888/metrics | grep "otelcol_exporter_sent"
  
  immediate_mitigation:
    - 增加队列大小(queue_size)
    - 增加消费者数量(num_consumers)
    - 启用持久化存储
    - 启用数据采样(临时降低数据量)
  
  root_cause_resolution:
    slow_backend:
      problem: 后端处理速度慢
      solutions:
        - 优化后端性能
        - 增加后端实例
        - 使用负载均衡
    
    network_congestion:
      problem: 网络拥塞
      solutions:
        - 增加带宽
        - 启用压缩
        - 调整超时设置
    
    data_spike:
      problem: 数据量突发增长
      solutions:
        - 实施流控
        - 增加Collector实例
        - 配置采样
```

### 3.4 Processor故障

```yaml
# Processor故障诊断
processor_failure_diagnosis:
  common_failures:
    memory_limiter_triggered:
      symptoms:
        - 日志: "Memory limiter processor is refusing data"
        - 数据被拒绝
        - 内存使用接近限制
      solution: |
        1. 检查limit_mib设置是否合理
        2. 检查spike_limit_mib是否足够
        3. 增加Collector内存
        4. 优化批处理大小
    
    batch_processor_stuck:
      symptoms:
        - 数据延迟增加
        - 批处理超时
        - CPU使用率异常
      solution: |
        1. 减小timeout值
        2. 调整send_batch_size
        3. 检查下游Exporter状态
    
    resource_processor_error:
      symptoms:
        - 属性未按预期添加
        - 处理器报错
      solution: |
        1. 检查属性配置语法
        2. 验证from_attribute是否存在
        3. 检查条件表达式
    
    filter_processor_too_aggressive:
      symptoms:
        - 数据量突然减少
        - 预期数据缺失
      solution: |
        1. 检查filter条件
        2. 验证正则表达式
        3. 调整过滤规则
```

### 4.1 CPU使用率过高

```bash
#!/bin/bash
# CPU问题诊断脚本

echo "=== CPU使用率诊断 ==="

echo "=== 诊断完成 ==="
echo "建议:"
echo "- 如果批处理过大，减小send_batch_size"
```

### 4.2 内存优化诊断

```yaml
# 内存优化检查清单
memory_optimization_checklist:
  configuration_review:
    - check: Memory Limiter配置
      command: grep -A 3 "memory_limiter" /etc/otel/config.yaml
      recommended: limit_mib = container_limit * 0.8
    
    - check: Batch处理器配置
      command: grep -A 5 "batch:" /etc/otel/config.yaml
      recommended: send_batch_size < 10000
    
    - check: 队列配置
      command: grep -A 5 "sending_queue" /etc/otel/config.yaml
      recommended: queue_size * avg_item_size < limit_mib * 0.2
  
  monitoring_items:
    - metric: otelcol_process_memory_rss
      alert_threshold: > 80% of limit
    
    - metric: otelcol_process_runtime_heap_alloc_bytes
      alert_threshold: > 70% of limit
    
    - metric: otelcol_queue_size
      alert_threshold: > 90% of capacity
  
  optimization_actions:
    action_1_reduce_batch_size:
      description: 减小批处理大小
      impact: 减少内存占用，增加请求次数
      config: |
        batch:
          send_batch_size: 1024  # 从8192减少
          timeout: 500ms         # 减少等待时间
    
    action_2_enable_persistence:
      description: 启用持久化队列
      impact: 降低内存压力，增加磁盘I/O
      config: |
        sending_queue:
          persistent_storage:
            enabled: true
            directory: /var/lib/otel/queue
    
    action_3_reduce_queue_size:
      description: 减小队列大小
      impact: 降低内存占用，可能增加数据丢失风险
      config: |
        sending_queue:
          queue_size: 5000  # 从10000减少
```

## 5. 网络与连接问题

### 5.1 TLS/SSL连接问题

```yaml
# TLS问题诊断
tls_connection_diagnosis:
  common_errors:
    certificate_verify_failed:
      error: "x509: certificate signed by unknown authority"
      diagnosis:
        - check_ca_certificate: |
            openssl s_client -connect backend:4317 -CAfile /etc/otel/certs/ca.crt
        - verify_certificate_chain: |
            openssl verify -CAfile ca.crt server.crt
      solutions:
        - 确保证书链完整
        - 更新CA证书
        - 检查证书有效期
    
    hostname_mismatch:
      error: "x509: certificate is valid for X, not Y"
      diagnosis:
        check_san: |
          openssl x509 -in server.crt -text | grep DNS
      solutions:
        - 更新证书包含正确的主机名
        - 使用server_name_override
        - 确保证书的SAN包含所有访问地址
    
    tls_version_mismatch:
      error: "tls: protocol version not supported"
      diagnosis:
        check_version: |
          openssl s_client -connect backend:4317 -tls1_3
      solutions:
        - 统一TLS版本配置
        - 禁用不安全的TLS版本
  
  debugging_commands:
    test_connection: |
      openssl s_client -connect backend:4317 \
        -cert /etc/otel/certs/client.crt \
        -key /etc/otel/certs/client.key \
        -CAfile /etc/otel/certs/ca.crt
    
    check_certificate_expiry: |
      openssl x509 -in server.crt -noout -dates
    
    dump_tls_handshake: |
      GRPC_GO_LOG_VERBOSITY_LEVEL=99 \
      GRPC_GO_LOG_SEVERITY_LEVEL=info \
      otelcol-contrib --config=/etc/otel/config.yaml
```

## 8. 紧急恢复方案

### 8.1 快速回滚方案

```bash
#!/bin/bash
# 紧急回滚脚本

BACKUP_DIR="/backup/otel/$(date +%Y%m%d%H%M%S)"
CURRENT_CONFIG="/etc/otel/config.yaml"

echo "=== OTLP紧急回滚脚本 ==="

# 1. 备份当前配置
echo "[1] 备份当前配置到 $BACKUP_DIR"
mkdir -p $BACKUP_DIR
cp $CURRENT_CONFIG $BACKUP_DIR/

# 2. 停止Collector
echo "[2] 停止Collector服务"
systemctl stop otelcol-contrib

# 3. 恢复上一版本配置
echo "[3] 恢复上一版本配置"
cp /etc/otel/config.yaml.backup $CURRENT_CONFIG

# 4. 验证配置
echo "[4] 验证配置"
otelcol-contrib validate --config=file:$CURRENT_CONFIG

if [ $? -eq 0 ]; then
    echo "[5] 启动Collector"
    systemctl start otelcol-contrib
    
    echo "[6] 检查状态"
    sleep 5
    systemctl status otelcol-contrib --no-pager
    curl -s http://localhost:13133/health
    
    echo "=== 回滚完成 ==="
else
    echo "配置验证失败，请手动检查"
    exit 1
fi
```

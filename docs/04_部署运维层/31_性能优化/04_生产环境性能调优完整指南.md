## 2. 采样策略优化

### 2.1 采样策略对比

| 采样类型 | 阶段 | CPU节省 | 数据保留 | 适用场景 |
|----------|------|---------|----------|----------|
| **AlwaysOn** | - | 0% | 100% | 开发调试 |
| **Probabilistic** | Head | 90% | 10% | 基准负载 |
| **RateLimiting** | Head | 85% | 自适应 | 突发流量 |
| **ParentBased** | Head | 90% | 一致 | 分布式追踪 |
| **TailBased** | Tail | 70% | 智能 | 生产环境 |

### 2.2 生产级采样配置

```yaml
# 推荐: Head + Tail 组合采样
processors:
  # 第一层: Head-Based概率采样 (减少90%数据)
  probabilistic_sampler:
    sampling_percentage: 10.0
    hash_seed: 22
  
  # 第二层: Tail-Based智能采样 (保留关键数据)
  tail_sampling:
    decision_wait: 10s          # 等待时间窗口
    num_traces: 100000          # 最大追踪数
    expected_new_traces_per_sec: 1000
    
    policies:
      # 策略1: 保留所有错误 (100%)
      - name: errors
        type: status_code
        status_code:
          status_codes: [ERROR]
      
      # 策略2: 保留慢请求 (P99>1s)
      - name: slow_requests
        type: latency
        latency:
          threshold_ms: 1000
      
      # 策略3: 概率采样 (5%)
      - name: probabilistic
        type: probabilistic
        probabilistic:
          sampling_percentage: 5
      
      # 策略4: 特定服务全采
      - name: critical_services
        type: string_attribute
        string_attribute:
          key: service.name
          values:
            - payment-service
            - auth-service
          enabled_regex_matching: false
```

## 3. 批处理优化

### 3.1 Batch Processor配置

```yaml
processors:
  batch:
    # 时间触发
    timeout: 1s                 # 最长等待时间
    
    # 大小触发
    send_batch_size: 1024       # 批次大小 (spans/metrics/logs)
    send_batch_max_size: 2048   # 最大批次大小
    
    # 元数据键 (多租户场景)
    metadata_keys:
      - tenant.id
      - service.name
```

### 3.2 批处理调优参数

| 场景 | timeout | send_batch_size | 效果 |
|------|---------|-----------------|------|
| 低延迟优先 | 100ms | 256 | 低延迟, 高CPU |
| 平衡模式 | 1s | 1024 | 平衡 |
| 吞吐优先 | 10s | 8192 | 高吞吐, 高延迟 |

### 3.3 批处理性能测试

```bash
# 测试不同批次大小的性能
for size in 256 512 1024 2048 4096 8192; do
  echo "Testing batch size: $size"
  
  # 修改配置
  yq e ".processors.batch.send_batch_size = $size" -i config.yaml
  
  # 启动Collector并记录性能
  otelcol --config config.yaml &
  
  # 运行负载测试
  loadtest --rate 10000 --duration 60s
  
  # 收集指标
  curl -s localhost:8888/metrics | grep 
    "otelcol_processor_batch_"
  
  kill %1
done
```

## 4. 内存管理

### 4.1 Memory Limiter配置

```yaml
processors:
  memory_limiter:
    # 检查间隔
    check_interval: 1s
    
    # 内存限制 (Mib)
    limit_mib: 1500
    
    # 尖峰限制 (应对突发流量)
    spike_limit_mib: 300
    
    # 限制策略
    limit_percentage: 75
    spike_limit_percentage: 15
```

### 4.2 内存优化策略

```yaml
processors:
  # 1. 属性过滤 (减少数据大小)
  attributes/filter:
    actions:
      - key: http.request.body
        action: delete
      - key: db.statement
        action: delete
      - key: exception.stacktrace
        action: delete
  
  # 2. 资源属性复用
  resource:
    attributes:
      - key: service.name
        from_attribute: service.name
        action: upsert
  
  # 3. 数据裁剪
  transform/trim:
    trace_statements:
      - context: span
        statements:
          # 限制属性值长度
          - set(attributes["http.url"], Substring(attributes["http.url"], 0, 256))
```

### 4.3 内存监控

```yaml
# 内存使用率告警
exporters:

processors:
```

## 5. 并发优化

### 5.1 并发配置

```yaml
exporters:
  otlp:
    endpoint: backend:4317
    
    # 发送队列
    sending_queue:
      enabled: true
```

## 6. 压缩优化

### 6.1 压缩算法选择

| 算法 | CPU开销 | 压缩率 | 适用场景 |
|------|---------|--------|----------|
| none | 0% | 0% | 内网低延迟 |
| gzip | 中 | 70% | 通用 |
| zstd | 低 | 75% | 高吞吐 |
| snappy | 很低 | 50% | 实时性要求 |

### 6.2 压缩配置

```yaml
exporters:
  otlp:
    endpoint: backend:4317
    compression: zstd           # 推荐zstd
    
  otlphttp:
    endpoint: https://backend:4318
    compression: gzip
    headers:
      Content-Encoding: gzip
```

## 7. 生产环境配置模板

### 7.1 高吞吐场景 (>100K spans/sec)

```yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
        max_recv_msg_size_mib: 64
        max_concurrent_streams: 1000

processors:
  memory_limiter:
    check_interval: 1s
    limit_mib: 4000
    spike_limit_mib: 800
  
  # 第一层采样: 10%
  probabilistic_sampler:
    sampling_percentage: 10
  
  # 智能尾采样
  tail_sampling:
    decision_wait: 5s
    num_traces: 500000
    expected_new_traces_per_sec: 10000
    policies:
      - name: errors
        type: status_code
        status_code: {status_codes: [ERROR]}
      - name: slow
        type: latency
        latency: {threshold_ms: 500}
      - name: probabilistic
        type: probabilistic
        probabilistic: {sampling_percentage: 5}
  
  # 大批次处理
  batch:
    timeout: 5s
    send_batch_size: 8192
    send_batch_max_size: 16384
    metadata_keys: [tenant.id]

exporters:
  otlp:
    endpoint: backend:4317
    compression: zstd
    sending_queue:
      enabled: true
      num_consumers: 20
      queue_size: 50000
    retry_on_failure:
      enabled: true
      max_elapsed_time: 300s

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [memory_limiter, probabilistic_sampler, tail_sampling, batch]
      exporters: [otlp]
```

### 7.2 低延迟场景 (<10ms开销)

## 9. 故障排查

### 9.1 性能问题诊断

```bash
# 1. 检查资源使用
curl -s localhost:8888/metrics | grep 
  -E "(process_cpu_seconds|process_memory|exporter_send_failed)"

# 2. 采样率验证
curl -s localhost:8888/metrics | grep 
  "tail_sampling"

# 3. 批处理效率
curl -s localhost:8888/metrics | grep 
  "processor_batch"
```

### 9.2 常见问题解决

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 内存溢出 | 批次过大 | 降低batch size |
| CPU过高 | 采样不足 | 增加采样率 |
| 数据丢失 | 队列满 | 增加queue_size |
| 延迟高 | 压缩耗时 | 改用snappy |

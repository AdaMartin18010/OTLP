# Performance 规范合规检查清单

> **标准来源**: OpenTelemetry Specification v1.56.0 — Performance
> **核心原则**: 可观测性库**不得**阻塞最终用户应用，**不得**消耗无界内存资源
> **适用范围**: SDK 实现者、框架开发者、平台工程师

---

## 一、核心原则总览

OpenTelemetry 官方 Performance 规范定义了所有客户端实现（API + SDK）必须遵守的三条铁律：

```text
┌─────────────────────────────────────────────────────────────┐
│  铁律一：默认情况下，库不得阻塞最终用户应用                      │
│  铁律二：库不得消耗无界内存资源                                │
│  铁律三：Shutdown 和显式 Flush 允许阻塞，但必须可配置超时        │
└─────────────────────────────────────────────────────────────┘
```

任何声称兼容 OpenTelemetry 规范的实现，都必须通过以下检查清单的验证。

---

## 二、非阻塞性检查清单

### 2.1 API 层非阻塞要求

| 检查项 | 要求 | 验证方法 |
|--------|------|---------|
| Span 创建 | `startSpan()` 不得触发网络 I/O、磁盘 I/O 或锁等待 | 单元测试：在 1ms 内必须返回 |
| 属性设置 | `setAttribute()` 不得触发序列化或远程调用 | 微基准测试：单次调用 < 100ns |
| Context 绑定 | `makeCurrent()` 不得涉及线程同步原语（在 no-op 路径上） | 代码审查 + 基准测试 |
| Baggage 设置 | `setBaggage()` 不得触发传播或导出 | 微基准测试 |
| Metrics 记录 | `record()` / `add()` 不得阻塞等待后端确认 | 异步确认机制审查 |

### 2.2 SDK 层非阻塞要求

| 检查项 | 要求 | 验证方法 |
|--------|------|---------|
| Span 导出 | Exporter 的 `export()` 调用应该是异步或带超时的 | 代码审查：确认使用线程池/异步 I/O |
| 批处理队列 | Batch Span Processor 的队列满时不得阻塞业务线程 | 压测：队列满时 `onEnd()` 仍应在 < 1ms 返回 |
| 指标上报 | Metrics 的周期性上报必须在独立线程/协程执行 | 线程模型审查 |
| 日志导出 | Logs 的导出不得阻塞应用日志写入路径 | 异步队列审查 |

### 2.3 阻塞操作的例外与规范

以下操作**允许**阻塞，但必须满足后续约束：

| 操作 | 允许条件 |
|------|---------|
| `shutdown()` | 必须暴露用户可配置的超时参数 |
| `flush()` | 必须暴露用户可配置的超时参数 |
| 同步 Exporter | 如果使用同步导出，必须在独立线程执行，不占用业务线程 |

## 三、内存边界检查清单

### 3.1 有界内存原则

OpenTelemetry 实现**必须**确保在以下极端情况下，内存使用仍有上限：

- 应用持续产生大量 Span（如每秒百万级）
- 网络中断导致导出堆积
- 应用长时间运行不重启
- 多线程高并发场景

### 3.2 各组件内存限制检查

| 组件 | 内存限制机制 | 检查点 |
|------|-------------|--------|
| Batch Span Processor | 队列容量上限 + 满队列丢弃策略 | 配置项 `maxQueueSize` 是否存在且默认有限 |
| Batch Log Record Processor | 同上 | 同上 |
| Metrics 累积器 | 时间窗口 + 属性基数限制 | 是否支持 `cardinalityLimit` |
| Context 存储 | 基于作用域的自动释放 | 确保 `Scope.close()` 被调用 |
| Baggage | 总大小限制（W3C 建议 8192 bytes） | 传播前是否检查大小 |

### 3.3 背压与丢弃策略

当系统负载超过处理能力时，实现**必须**提供明确的选择：

```text
策略 A：防止信息丢失（保留所有数据，可能消耗大量内存）
        └── 适用于：内存充足、数据不可丢失的场景
        └── 风险：可能引发 OOM

策略 B：防止阻塞（在超载时丢弃部分数据，并输出警告日志）
        └── 适用于：内存敏感、可接受部分数据丢失的场景
        └── 要求：
            ├── 提供可配置的丢弃阈值
            ├── 提供指标（如 effective sampling ratio）反映实际丢弃率
            └── 输出警告日志通知用户丢弃的开始与恢复
```

**检查清单**:

- [ ] 实现支持配置队列满时的行为（阻塞 vs 丢弃）
- [ ] 默认行为是"防止阻塞"（避免无界内存增长）
- [ ] 提供指标或日志，让用户感知到数据丢弃事件

## 四、日志资源管理检查清单

### 4.1 日志量控制

日志（尤其是应用日志通过 Bridge 导入 OpenTelemetry）是内存消耗的高风险点：

| 检查项 | 要求 |
|--------|------|
| 日志过滤 | 必须提供机制，允许用户过滤要发送到 Exporter 的日志级别/模式 |
| 默认行为 | 默认保留所有日志（防止信息丢失），但文档必须明确警告内存风险 |
| 批量导出 | 日志必须批量导出，不得逐条触发网络请求 |

### 4.2 文档要求

实现文档**必须**包含以下警告：

> "如果您的应用产生大量日志，OpenTelemetry 客户端默认会尝试保留并导出所有日志，这可能导致显著的内存和带宽消耗。建议通过配置日志过滤器，仅将需要远程分析的高价值日志发送到 Exporter。"

## 五、并发安全与线程安全

### 5.1 API 并发安全

| 对象 | 线程安全要求 |
|------|-------------|
| `TracerProvider` | 线程安全，可在多线程共享 |
| `Tracer` | 线程安全 |
| `Span` | 非线程安全，每个 Span 应由单个线程操作（或外部同步） |
| `MeterProvider` | 线程安全 |
| `Meter` / `Instrument` | 线程安全 |

### 5.2 SDK 并发安全

| 组件 | 线程安全要求 |
|------|-------------|
| `BatchSpanProcessor` | 线程安全，`onEnd()` 可被多线程并发调用 |
| `SimpleSpanProcessor` | 线程安全，但同步导出可能引入竞争 |
| `PeriodicMetricReader` | 采集与导出必须在独立线程执行 |

## 六、Shutdown 与 Flush 规范

### 6.1 Shutdown 行为

```java
// 规范示例：可配置超时的 Shutdown
tracerProvider.shutdown().join(10, TimeUnit.SECONDS); // Java
// 或
await tracerProvider.shutdown({ timeoutMillis: 10000 }); // JS
```

| 检查项 | 要求 |
|--------|------|
| 超时参数 | 必须支持用户配置超时时间 |
| 默认超时 | 应有合理的默认值（如 5-30 秒） |
| 超时后行为 | 超时后强制放弃未导出的数据，不得无限等待 |
| 重复调用 | `shutdown()` 多次调用应该是安全的（幂等） |

### 6.2 Flush 行为

| 检查项 | 要求 |
|--------|------|
| 显式 Flush | 如果支持显式 Flush，必须支持可配置超时 |
| 触发时机 | Flush 通常由用户或测试代码触发，不应在常规路径频繁调用 |

## 七、性能基准测试模板

### 7.1 必测场景

实现者应提供以下基准测试结果：

```text
测试场景 1：No-op API 调用开销
├── 目标：SDK 未安装时，startSpan + setAttribute + end 的总耗时
├── 基准：单次调用 < 50ns（C++/Java/Go），< 200ns（动态语言）
└── 工具：JMH（Java）、Criterion（Rust）、BenchmarkDotNet（.NET）

测试场景 2：已采样 Span 创建开销
├── 目标：SDK 安装、AlwaysOn 采样、Batch Processor
├── 基准：单次 startSpan + end < 1μs
└── 注意：不包括导出耗时

测试场景 3：未采样 Span 创建开销
├── 目标：SDK 安装、AlwaysOff 采样
├── 基准：单次 startSpan + end < 100ns
└── 验证：未采样路径应尽可能轻量

测试场景 4：内存增长测试
├── 目标：持续创建 Span，网络中断导致导出失败
├── 基准：内存使用应在配置队列大小对应的上限内稳定，不得无限增长
└── 时长：至少运行 10 分钟或 1000 万次操作

测试场景 5：并发扩展性
├── 目标：多线程（如 32/64/128 线程）并发创建 Span
├── 基准：吞吐量应随线程数线性或接近线性增长，无明显锁竞争瓶颈
└── 工具：perf / async-profiler 检查锁热点
```

### 7.2 报告格式

性能报告**应该**包含：

- 测试环境（CPU、内存、语言运行时版本）
- OpenTelemetry SDK 版本
- 每个场景的平均耗时、P99 耗时、吞吐量
- 内存使用峰值与稳定值
- 与之前版本的对比（如果有）

## 八、合规自评检查表

在发布 SDK 或框架集成之前，请逐项确认：

### API 层

- [ ] 所有 API 方法在 no-op 模式下不抛异常
- [ ] no-op 模式下属性设置和 Span 创建无显著性能开销
- [ ] `Span.current()` 在无上下文时返回有效的 no-op Span

### SDK 层

- [ ] Batch Processor 队列容量可配置且有默认值
- [ ] 队列满时默认行为是丢弃（而非阻塞业务线程）
- [ ] 丢弃事件有日志警告或指标上报
- [ ] Exporter 在独立线程/异步执行
- [ ] Shutdown 支持可配置超时
- [ ] Flush 支持可配置超时（如果提供显式 Flush）

### Metrics

- [ ] Metrics 的累积/聚合不阻塞业务线程
- [ ] 支持属性基数限制（cardinality limit）
- [ ] 周期性上报在独立线程执行

### Logs

- [ ] 提供日志过滤配置
- [ ] 文档明确警告大量日志的内存风险
- [ ] 日志批量导出，不逐条发送

### 文档

- [ ] 文档说明了性能特征和内存使用预期
- [ ] 文档提供了调优建议（队列大小、批量大小、超时设置）

## 九、常见反模式

| 反模式 | 危害 | 正确做法 |
|--------|------|---------|
| 在 API 层执行同步网络请求 | 阻塞业务线程 | 网络操作限制在 SDK Exporter 层，且异步执行 |
| 使用无界队列缓存待导出数据 | OOM 风险 | 使用有界队列，满时丢弃 |
| Span 未结束导致内存泄漏 | 持续增长 | 确保 try-finally 或作用域模式正确关闭 Span |
| 高频创建 Tracer 实例 | 资源浪费 | Tracer 应复用，按 instrumentation scope 创建 |
| 忽略未采样路径的优化 | 即使不采样也消耗过多 CPU | 未采样路径应尽早返回，避免属性序列化等操作 |

---

> **总结**: Performance 规范的本质是**信任契约**——用户信任 OpenTelemetry 不会拖垮他们的应用。通过本检查清单的系统性验证，可以确保实现对得起这份信任。

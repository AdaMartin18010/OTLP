# 语义约定迁移指南 - v1.40.0新旧约定迁移

> **语义约定版本**: v1.40.0  
> **迁移策略**: 双发射(Double Emission)  
> **最后更新**: 2026-03-17  

---

## 1. 迁移概述

Semantic Conventions v1.40.0对多个领域的属性进行了重命名和重组。本指南提供完整的迁移路径。

### 1.1 影响范围

| 领域 | 变更类型 | 稳定状态 |
|------|----------|----------|
| **HTTP** | 属性重命名 | Spans ✅ Stable |
| **Database** | 属性重命名 | ✅ Stable |
| **RPC** | 属性重命名 | Spans ✅ Stable |
| **Messaging** | 属性重命名 | Spans ✅ Stable |
| **Network** | 属性重命名 | ✅ Stable |

### 1.2 迁移时间线

```
2024 H1: 新约定发布 (opt-in)
2024 H2: 双发射支持 (新旧同时发送)
2025 H1: 新约定默认 (旧约定废弃)
2025 H2: 旧约定移除 (breaking change)
```

## 2. HTTP约定迁移

### 2.1 属性重命名映射

| 旧属性名 (Deprecated) | 新属性名 (Stable) | 示例值 |
|----------------------|-------------------|--------|
| `http.method` | `http.request.method` | `GET`, `POST` |
| `http.url` | `url.full` | `https://example.com/path?query=1` |
| `http.target` | `url.path` + `url.query` | `/path` + `?query=1` |
| `http.host` | `server.address` + `server.port` | `example.com` + `443` |
| `http.scheme` | `url.scheme` | `https` |
| `http.status_code` | `http.response.status_code` | `200`, `404` |
| `http.flavor` | `network.protocol.name` | `http`, `http/2` |
| `http.user_agent` | `user_agent.original` | `Mozilla/5.0...` |
| `http.request_content_length` | `http.request.body.size` | `1024` |
| `http.response_content_length` | `http.response.body.size` | `2048` |

### 2.2 完整映射示例

**旧约定 (v1.20.0及之前)**:
```json
{
  "http.method": "POST",
  "http.url": "https://api.example.com/v1/users?name=john",
  "http.host": "api.example.com",
  "http.scheme": "https",
  "http.status_code": 201,
  "http.flavor": "http/2",
  "http.request_content_length": 256
}
```

**新约定 (v1.40.0)**:
```json
{
  "http.request.method": "POST",
  "url.full": "https://api.example.com/v1/users?name=john",
  "url.scheme": "https",
  "url.path": "/v1/users",
  "url.query": "name=john",
  "server.address": "api.example.com",
  "server.port": 443,
  "http.response.status_code": 201,
  "network.protocol.name": "http",
  "network.protocol.version": "2",
  "http.request.body.size": 256
}
```

### 2.3 服务端 vs 客户端差异

```yaml
# 客户端 (Client)
client.address: "192.168.1.100"    # 本机地址
client.port: 54321
server.address: "api.example.com"  # 目标地址
server.port: 443

# 服务端 (Server)
client.address: "192.168.1.100"    # 请求来源
client.port: 54321
server.address: "api.internal"     # 本机服务地址
server.port: 8080
```

## 3. Database约定迁移

### 3.1 属性重命名映射

| 旧属性名 | 新属性名 | 说明 |
|----------|----------|------|
| `db.system` | `db.system.name` | 数据库类型 |
| `db.connection_string` | - | **移除** - 包含敏感信息 |
| `db.user` | `db.user.name` | 用户名 |
| `db.statement` | `db.query.text` | SQL语句 |
| `db.operation` | `db.operation.name` | 操作类型 |
| `db.sql.table` | `db.collection.name` | 表/集合名 |

### 3.2 新增属性

| 新属性名 | 说明 | 示例 |
|----------|------|------|
| `db.query.summary` | 参数化查询摘要 | `SELECT * FROM users WHERE id = ?` |
| `db.response.returned_rows` | 返回行数 | `100` |
| `db.response.status_code` | 数据库状态码 | `42P01` (PostgreSQL) |

### 3.3 数据库特定映射

#### MySQL
```yaml
# 旧约定
db.system: mysql
db.statement: "SELECT * FROM users WHERE id = 123"

# 新约定
db.system.name: mysql
db.query.text: "SELECT * FROM users WHERE id = 123"
db.query.summary: "SELECT * FROM users WHERE id = ?"
db.collection.name: users
```

#### Redis
```yaml
# 旧约定
db.system: redis
db.statement: "GET user:123"

# 新约定
db.system.name: redis
db.operation.name: GET
db.query.text: "GET user:123"
db.query.summary: "GET ?"
```

#### MongoDB
```yaml
# 旧约定
db.system: mongodb
db.operation: find
db.mongodb.collection: users

# 新约定
db.system.name: mongodb
db.operation.name: find
db.collection.name: users
db.query.text: '{"find": "users", "filter": {"age": {"$gt": 18}}}'
```

## 4. RPC约定迁移

### 4.1 属性重命名映射

| 旧属性名 | 新属性名 |
|----------|----------|
| `rpc.system` | `rpc.system.name` |
| `rpc.service` | `rpc.service.name` |
| `rpc.method` | `rpc.method.name` |
| `rpc.grpc.status_code` | `rpc.grpc.status_code` (不变) |

### 4.2 gRPC特定映射

```yaml
# 旧约定
rpc.system: grpc
rpc.service: helloworld.Greeter
rpc.method: SayHello
rpc.grpc.status_code: 0

# 新约定
rpc.system.name: grpc
rpc.service.name: helloworld.Greeter
rpc.method.name: SayHello
rpc.grpc.status_code: 0  # 不变
```

## 5. Network约定迁移

### 5.1 属性重命名映射

| 旧属性名 | 新属性名 | 说明 |
|----------|----------|------|
| `net.peer.name` | `server.address` (client) / `client.address` (server) | 对端地址 |
| `net.peer.port` | `server.port` (client) / `client.port` (server) | 对端端口 |
| `net.host.name` | `server.address` | 服务地址 |
| `net.host.port` | `server.port` | 服务端口 |
| `net.transport` | `network.transport` | 传输协议 |
| `net.protocol.name` | `network.protocol.name` | 应用协议 |
| `net.protocol.version` | `network.protocol.version` | 协议版本 |
| `net.sock.peer.addr` | `network.peer.address` | 对端Socket地址 |
| `net.sock.host.addr` | `network.local.address` | 本端Socket地址 |

## 6. 迁移策略

### 6.1 双发射模式 (推荐)

使用环境变量启用双发射模式，同时发送新旧两种约定：

```bash
# 同时发送HTTP新旧约定
export OTEL_SEMCONV_STABILITY_OPT_IN="http/dup"

# 同时发送HTTP和Database新旧约定
export OTEL_SEMCONV_STABILITY_OPT_IN="http/dup,database/dup"

# 仅使用新约定（默认未来行为）
export OTEL_SEMCONV_STABILITY_OPT_IN=""
```

### 6.2 SDK配置示例

#### Java
```java
// 通过系统属性
System.setProperty("otel.semconv-stability.opt-in", "http/dup");

// 或通过环境变量
// OTEL_SEMCONV_STABILITY_OPT_IN=http/dup
```

#### Python
```python
import os
os.environ["OTEL_SEMCONV_STABILITY_OPT_IN"] = "http/dup,database/dup"
```

#### Go
```go
os.Setenv("OTEL_SEMCONV_STABILITY_OPT_IN", "http/dup")
```

#### Node.js
```javascript
process.env.OTEL_SEMCONV_STABILITY_OPT_IN = 'http/dup';
```

### 6.3 后端适配建议

```yaml
# 在Collector中处理新旧约定共存
processors:
  transform:
    trace_statements:
      # 如果新属性不存在，从旧属性复制
      - set(attributes["http.request.method"], attributes["http.method"]) 
        where attributes["http.request.method"] == nil
      
      - set(attributes["http.response.status_code"], attributes["http.status_code"])
        where attributes["http.response.status_code"] == nil
```

## 7. 废弃指标列表 (v1.40.0)

### 7.1 RPC指标

| 废弃指标 | 说明 | 替代方案 |
|----------|------|----------|
| `rpc.client.request.size` | 客户端请求大小 | 暂无直接替代 |
| `rpc.client.response.size` | 客户端响应大小 | 暂无直接替代 |
| `rpc.server.request.size` | 服务端请求大小 | 暂无直接替代 |
| `rpc.server.response.size` | 服务端响应大小 | 暂无直接替代 |

### 7.2 事件

| 废弃事件 | 说明 | 替代方案 |
|----------|------|----------|
| `rpc.message` | RPC消息事件 | 暂无直接替代 |

### 7.3 属性

| 废弃属性 | 说明 | 替代方案 |
|----------|------|----------|
| `error.message` | 错误消息 | 使用领域特定错误属性 |
| `rpc.message.type` | RPC消息类型 | 暂无直接替代 |
| `rpc.message.compressed_size` | 压缩后大小 | 暂无直接替代 |
| `rpc.message.uncompressed_size` | 未压缩大小 | 暂无直接替代 |

## 9. 向后兼容性保证

### 9.1 版本兼容性矩阵

| SDK版本 | 旧约定支持 | 新约定支持 | 双发射支持 |
|---------|-----------|-----------|-----------|
| v1.20.0 | ✅ 默认 | ❌ 不支持 | ❌ 不支持 |
| v1.25.0 | ✅ 默认 | ✅ opt-in | ❌ 不支持 |
| v1.30.0 | ✅ 默认 | ✅ opt-in | ✅ 支持 |
| v2.0.0+ | ❌ 移除 | ✅ 默认 | ❌ 不适用 |

### 9.2 迁移阶段建议

```
阶段1: 评估 (1-2周)
  └── 识别影响范围
  └── 测试新约定兼容性
  
阶段2: 双发射 (4-8周)
  └── 启用双发射模式
  └── 更新后端和仪表盘
  └── 监控新旧数据质量
  
阶段3: 切换 (2-4周)
  └── 后端切换默认使用新约定
  └── 禁用旧约定仪表盘
  
阶段4: 清理 (1-2周)
  └── 禁用双发射
  └── 移除旧约定支持
```

# 三大支柱对比与统一模型

> **适用读者**: 可观测性初学者到架构师
> **技术难度**: ⭐⭐ (初级到中级)
> **最后更新**: 2026-03-17

## 1. 三大支柱概述

### 1.1 可观测性定义

```
可观测性 (Observability):

从系统外部输出推断内部状态的能力

外部输出:
├── Metrics (指标)    → 系统状态快照
├── Logs (日志)       → 离散事件记录
└── Traces (链路)     → 请求路径追踪

可观测性 = Metrics + Logs + Traces (+ Profiles)
```

### 1.2 演进历程

```
可观测性演进:

第一代 (2000s): 仅Logs
└── 文本日志, grep分析

第二代 (2010s): Logs + Metrics
├── Metrics: Nagios, Zabbix
└── Logs: ELK Stack

第三代 (2015s): Logs + Metrics + Traces
├── Metrics: Prometheus
├── Logs: ELK/Loki
└── Traces: Zipkin, Jaeger

第四代 (2020s): OpenTelemetry统一
└── OTLP: Metrics + Logs + Traces 统一协议
```

## 2. 三大支柱深度对比

### 2.1 核心特性对比表

| 维度 | Metrics | Logs | Traces |
|------|---------|------|--------|
| **数据模型** | 数值型时间序列 | 文本/结构化事件 | 因果关系的Span树 |
| **数据量** | 低 (聚合后) | 高 (原始) | 中 (采样后) |
| **存储成本** | 低 | 高 | 中 |
| **查询复杂度** | 低 | 高 | 中 |
| **实时性** | 秒级 | 秒-分钟级 | 秒级 |
| **最佳用途** | 趋势/告警 | 调试/审计 | 性能/依赖 |

### 2.2 可视化对比

```
同一问题的三种视角:

场景: API错误率上升

Metrics视角:
┌─────────────────────────────────────┐
│  error_rate{service="api"}          │
│    ↑                                  │
│ 15%│      ╱╲                          │
│ 10%│     ╱  ╲                         │
│  5%│____╱    ╲___                     │
│  0%  10:00  10:05  10:10              │
│     时间序列趋势图                      │
└─────────────────────────────────────┘
→ 发现了问题 (What)

Logs视角:
┌─────────────────────────────────────┐
│  10:05:23 ERROR api Payment failed  │
│  10:05:24 ERROR api DB timeout      │
│  10:05:25 ERROR api Connection reset│
│  ...                                  │
└─────────────────────────────────────┘
→ 理解了问题详情 (Why)

Traces视角:
┌─────────────────────────────────────┐
│  Request-1234 (2.3s)                │
│  ├── gateway (10ms) ✓               │
│  ├── auth (50ms) ✓                  │
│  ├── payment (2000ms) ✗             │
│  │   └── db-query (1950ms) ✗        │
│  └── notification (240ms) ✓         │
└─────────────────────────────────────┘
→ 定位了问题位置 (Where)
```

## 3. Metrics详解

### 3.1 Metric类型

```python
# OpenTelemetry Metrics类型详解

from opentelemetry.metrics import (
    Counter,            # 累加值 (请求数)
    UpDownCounter,      # 可增可减 (连接数)
    Gauge,              # 瞬时值 (温度)
    Histogram,          # 分布统计 (延迟)
    ObservableGauge,    # 异步测量 (内存)
)

# 1. Counter - 单调递增
request_counter = meter.create_counter(
    "http_requests_total",
    description="Total HTTP requests"
)
request_counter.add(1, {"method": "GET", "status": "200"})

# 2. Histogram - 延迟分布
latency_histogram = meter.create_histogram(
    "http_request_duration_seconds",
    description="HTTP request latency"
)
latency_histogram.record(0.23, {"path": "/api/users"})

# 3. Gauge - 瞬时值
cpu_gauge = meter.create_gauge(
    "cpu_usage_percent",
    description="Current CPU usage"
)
cpu_gauge.set(75.5, {"core": "0"})
```

### 3.2 Metric聚合

```
聚合模型对比:

原始数据点 (每秒):
[100, 102, 98, 101, 99, 105, 97, 103]

不同聚合方式:
├── Counter: SUM → 805 (总计)
├── Gauge: LAST → 103 (最新)
├── Histogram:
│   ├── buckets: [0-100:3, 100-200:5]
│   ├── count: 8
│   ├── sum: 805
│   └── quantile(0.99): ~105
└── Summary: 类似Histogram但客户端计算
```

## 4. Logs详解

### 4.1 Log模型

```yaml
# OpenTelemetry LogRecord 结构
LogRecord:
  # 时间信息
  timestamp: 2026-03-17T14:30:00Z
  observed_timestamp: 2026-03-17T14:30:00.001Z

  # 严重性
  severity_number: 17    # ERROR
  severity_text: "ERROR"

  # 消息体
  body: "Database connection failed"

  # 结构化属性
  attributes:
    db.system: postgresql
    db.name: orders
    error.type: ConnectionTimeout

  # 追踪关联
  trace_id: 4bf92f3577b34da6a3ce929d0e0e4736
  span_id: 00f067aa0ba902b7

  # 资源信息
  resource:
    service.name: payment-service
    host.name: prod-server-01
```

### 4.2 Log与Trace关联

```python
# 在日志中注入Trace上下文
import logging
from opentelemetry.trace import get_current_span

class TraceContextFilter(logging.Filter):
    """自动添加Trace ID到日志"""

    def filter(self, record):
        span = get_current_span()
        if span:
            context = span.get_span_context()
            record.trace_id = format(context.trace_id, '032x')
            record.span_id = format(context.span_id, '016x')
        else:
            record.trace_id = "0"
            record.span_id = "0"
        return True

# 配置
logger = logging.getLogger("myapp")
logger.addFilter(TraceContextFilter())

# 使用
logger.error("Payment failed")  # 自动包含trace_id

# 输出:
# 2026-03-17 ERROR [trace_id=4bf92f...] Payment failed
```

## 5. Traces详解

### 5.1 Span模型

```protobuf
// OTLP Span 结构
message Span {
  bytes trace_id = 1;
  bytes span_id = 2;
  string trace_state = 3;
  bytes parent_span_id = 4;

  string name = 5;
  SpanKind kind = 6;

  fixed64 start_time_unix_nano = 7;
  fixed64 end_time_unix_nano = 8;

  repeated KeyValue attributes = 9;
  uint32 dropped_attributes_count = 10;

  repeated Event events = 11;
  uint32 dropped_events_count = 12;

  repeated Link links = 13;
  uint32 dropped_links_count = 14;

  Status status = 15;
}
```

### 5.2 Trace可视化

```
Trace: 用户下单流程

Span ID      Operation              Duration    Status
─────────────────────────────────────────────────────
[root]       POST /api/orders       245ms       OK
├── [s1]     validate_request        5ms        OK
├── [s2]     auth_check             20ms        OK
├── [s3]     create_order          100ms        OK
│   ├── [s3.1]  db_insert          80ms        OK
│   └── [s3.2]  cache_update       15ms        OK
├── [s4]     process_payment        80ms        OK
│   ├── [s4.1]  payment_gateway    60ms        OK
│   └── [s4.2]  update_status      15ms        OK
└── [s5]     send_notification      30ms        OK
```

## 6. OpenTelemetry统一模型

### 6.1 统一数据模型

```
OTLP统一模型:

ExportRequest
├── Resource                       # 通用资源信息
│   ├── service.name
│   ├── service.version
│   └── host.name
│
├── Scope (Instrumentation Scope)  # 库/组件信息
│   ├── name: "http-client"
│   └── version: "1.2.0"
│
└── Data
    ├── Metrics
    │   └── Metric [Gauge|Sum|Histogram|...]
    ├── Logs
    │   └── LogRecord
    └── Traces
        └── Span

共同点:
├── Resource属性共享
├── Context传播统一
├── 相同的导出机制
└── 统一的语义约定
```

### 6.2 关联与导航

```python
# 三大支柱的关联实现

# 1. Trace → Logs (通过Trace ID)
def process_order(order_id):
    with tracer.start_as_current_span("process_order") as span:
        # 日志自动包含Trace ID
        logger.info(f"Processing order {order_id}")

        # 添加Metric标签
        orders_total.add(1, {
            "customer_tier": get_tier(order_id)
        })

        # Span记录事件
        span.add_event("validation_complete", {
            "order_id": order_id
        })

# 2. Logs → Trace (通过LogRecord中的trace_id/span_id)
# 3. Metrics → Trace (通过Exemplar)
histogram.record(0.15, attributes={"path": "/api"})
# 高延迟点关联到具体Trace
```

### 6.3 统一查询语言

```sql
-- TraceQL: 统一查询Trace
{ span.http.status_code = 500 }

-- LogQL: 查询日志
{service="payment"} |= "error" | trace_id="4bf92f..."

-- PromQL: 查询Metrics
rate(http_requests_total{status="500"}[5m])

-- 关联查询 (Grafana Tempo + Loki + Prometheus)
-- 从Alert → Metric → Trace → Log 无缝跳转
```

## 7. 选型指南

### 7.1 使用场景决策树

```
需要监控什么?
    │
    ├── 系统健康/告警? ──→ Metrics
    │   └── 示例: CPU使用率、请求率、错误率
    │
    ├── 问题调试/审计? ──→ Logs
    │   └── 示例: 错误详情、用户行为、安全事件
    │
    └── 性能优化/依赖分析? ──→ Traces
        └── 示例: 延迟分布、服务依赖、瓶颈定位
    │
    ├── 全链路问题? ──→ Metrics + Traces + Logs
    │   └── 工作流程:
│       1. Metrics告警 (错误率>5%)
│       2. Traces定位 (payment服务)
│       3. Logs详情 (DB timeout)
```

### 7.2 数据量对比

```
典型生产环境 (1000 RPS):

Metrics:
├── 指标数量: ~1000个
├── 采集频率: 15s
├── 日数据量: ~5MB
└── 保留期: 15天

Logs:
├── 每请求: ~5条日志
├── 每条: ~500字节
├── 日数据量: ~200GB
└── 保留期: 7天

Traces (10%采样):
├── 每请求: ~20 spans
├── 每span: ~500字节
├── 日数据量: ~80GB
└── 保留期: 3天

总计: ~280GB/天
成本优化: Metrics全保留 + Logs/Trace分级存储
```

## 8. 最佳实践

### 8.1 统一字段规范

```yaml
# 跨信号统一字段
resource:
  # 服务标识
  service.name: "payment-service"
  service.namespace: "ecommerce"
  service.instance.id: "payment-7d9f4-01"
  service.version: "v2.3.1"

  # 部署信息
  deployment.environment: "production"
  host.name: "prod-k8s-node-05"

  # 通用属性 (Metrics/Logs/Traces一致)
  attributes:
    customer_id: "cust_12345"
    region: "ap-southeast-1"
```

### 8.2 关联策略

```python
# 实现三大支柱的深度关联

class UnifiedObservability:
    def __init__(self):
        self.tracer = trace.get_tracer(__name__)
        self.meter = metrics.get_meter(__name__)
        self.logger = logging.getLogger(__name__)

        # 创建带Exemplar的Histogram
        self.latency = self.meter.create_histogram(
            "request_latency",
            description="Request latency with trace exemplars"
        )

    def handle_request(self, request):
        with self.tracer.start_as_current_span("handle_request") as span:
            start = time.time()

            try:
                result = self.process(request)

                # 记录延迟 (自动关联当前Trace)
                duration = time.time() - start
                self.latency.record(duration)

                # 结构化日志
                self.logger.info(
                    "Request processed",
                    extra={
                        "duration": duration,
                        "request_id": request.id,
                        # Trace ID自动注入
                    }
                )

                return result

            except Exception as e:
                span.record_exception(e)
                span.set_status(StatusCode.ERROR)
                self.logger.exception("Request failed")
                raise
```

# AIOps 平台开源架构设计

> **核心目标**: 基于 OpenTelemetry 数据构建 AIOps 平台的开源参考架构，涵盖异常检测、根因分析、预测性维护和智能采样

### 1.2 核心设计原则

| 原则 | 说明 |
|------|------|
| **数据驱动** | 所有 AI 模型基于 OpenTelemetry 统一数据 |
| **可解释性** | AI 决策必须可追溯、可解释 |
| **实时性** | 异常检测延迟 < 30 秒 |
| **可扩展性** | 支持从单服务到万级节点的规模 |
| **开源优先** | 核心组件使用开源技术栈 |

## 二、数据管道设计

### 2.1 实时管道（异常检测）

```text
实时管道 (< 30s 延迟):

OTel SDK → Collector → Kafka/Pulsar → Stream Processor → Alert Manager
              │                              │
              │                              ├── 窗口聚合（5s tumbling window）
              │                              ├── 特征提取（延迟、错误率、QPS）
              │                              └── 模型推理（Isolation Forest / LSTM）
              │
              └── 批处理分支 → 数据湖（长期存储 / 训练数据）
```

### 2.2 批处理管道（模型训练）

```text
批处理管道（小时/天级）:

数据湖 (S3/GCS) → ETL (Spark/Flink) → 特征工程 → 模型训练 → 模型仓库
                                                      │
                                                      ├── 时序预测模型 (Prophet / ARIMA)
                                                      ├── 异常检测模型 (LSTM Autoencoder)
                                                      ├── 分类模型（故障类型识别）
                                                      └── 根因图模型（GNN / 贝叶斯网络）
```

## 三、核心模块设计

### 3.1 异常检测引擎

```text
异常检测架构:

输入: Metrics 时序数据（延迟、错误率、CPU、内存、QPS）

预处理:
├── 缺失值填充（线性插值 / 前向填充）
├── 归一化（Z-score / Min-Max）
├── 季节性分解（STL / Prophet）
└── 降维（PCA / 特征选择）

检测算法（多模型融合）:
├── 统计模型: 3-sigma, MAD, IQR（简单快速）
├── 机器学习: Isolation Forest, One-Class SVM
├── 深度学习: LSTM Autoencoder, Transformer
└── 集成决策: 投票 / 加权融合

输出:
├── 异常标签（0/1）
├── 异常分数（0-1）
├── 异常类型（点异常 / 上下文异常 / 集体异常）
└── 置信度
```

### 3.2 根因分析引擎

```text
根因分析架构:

输入: 异常事件 + Traces + Metrics + Logs

知识图谱构建:
├── 服务依赖图（从 Traces 提取调用关系）
├── 基础设施拓扑（从 Resource 属性提取）
├── 变更事件关联（从 CICD 语义约定提取）
└── 历史故障库（人工标注的历史 RCA 结果）

分析算法:
├── 基于追踪: 从异常 Span 向上追溯，找到首个异常父节点
├── 基于度量: 相关性分析，找到与异常指标最相关的上游指标
├── 基于图: PageRank / Random Walk 在依赖图上定位根因服务
└── 基于知识: 规则引擎匹配已知故障模式

输出:
├── 根因服务列表（按概率排序）
├── 根因证据链（Span → Metric → Log 关联）
├── 建议修复动作
└── 历史相似案例
```

### 3.3 智能采样引擎

```text
智能采样架构:

输入: 实时 Traces + 异常信号

采样策略:
├── 正常流量: 低采样率（1%）
├── 异常流量: 高采样率（100%）
├── 关键路径: 始终采样（支付、登录）
└── 长尾请求: 自适应采样（慢请求自动捕获）

算法:
├── 基于延迟: 超过 P99 的请求强制采样
├── 基于错误: 错误请求强制采样
├── 基于异常检测: 异常区域的流量提高采样率
└── 基于用户: VIP 用户请求优先采样

输出:
├── 采样决策（采样 / 不采样）
├── 采样率调整信号（反馈给 SDK）
└── 采样效果报告（覆盖率 vs 存储成本）
```

## 四、技术栈选型

| 层次 | 组件 | 选型 | 说明 |
|------|------|------|------|
| **数据采集** | OTel SDK | 官方多语言 SDK | 应用插桩 |
| **数据收集** | OTel Collector | 官方 Collector | 数据处理和路由 |
| **消息队列** | Kafka / Pulsar | Apache Kafka | 实时流数据缓冲 |
| **流处理** | Flink / Kafka Streams | Apache Flink | 实时异常检测 |
| **时序存储** | Prometheus / VictoriaMetrics | VictoriaMetrics | Metrics 存储和查询 |
| **追踪存储** | Jaeger / Tempo | Grafana Tempo | Traces 存储 |
| **日志存储** | Loki / Elasticsearch | Grafana Loki | Logs 存储 |
| **特征存储** | Feast / Redis | Redis | 实时特征服务 |
| **模型训练** | MLflow / Kubeflow | MLflow | 模型生命周期管理 |
| **深度学习** | PyTorch / TensorFlow | PyTorch | 时序预测和异常检测 |
| **图数据库** | Neo4j / Dgraph | Neo4j | 依赖关系图谱 |
| **可视化** | Grafana | Grafana + 自定义插件 | 仪表盘和告警 |

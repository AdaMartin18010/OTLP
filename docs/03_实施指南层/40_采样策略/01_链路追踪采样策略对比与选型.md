# 链路追踪采样策略对比与选型

> **适用场景**: 生产环境链路追踪采样策略设计
> **技术难度**: ⭐⭐⭐⭐ (中高级)
> **最后更新**: 2026-03-17

---

## 1. 采样策略概览

### 1.1 为什么需要采样

```
生产环境采样必要性:

成本维度:
├── 存储成本: 全量存储TB级/天
├── 网络带宽: 跨可用区传输费用
├── 计算资源: Collector处理开销
└── 后端成本: Trace存储系统

技术维度:
├── 性能影响: SDK overhead
├── 数据价值: 99.9%正常trace价值低
├── 检索效率: 海量数据检索困难
└── 可视化: 调用图过于密集

目标: 在10%采样下捕获90%问题
```

### 1.2 采样策略分类

| 策略 | 阶段 | 决策时机 | 适用场景 |
|------|------|----------|----------|
| **头部采样** | SDK端 | Span创建时 | 简单比例、低开销 |
| **尾部采样** | Collector | Trace完成时 | 异常检测、业务规则 |
| **自适应采样** | 动态 | 运行时调整 | 流量波动、成本敏感 |
| **混合采样** | 组合 | 多阶段 | 复杂生产环境 |

## 2. 头部采样 (Head Sampling)

### 2.1 实现原理

```python
# 头部采样示例
class HeadSamplingSDK:
    """SDK端头部采样实现"""

    def __init__(self, strategy="probabilistic", rate=0.1):
        self.strategy = strategy
        self.rate = rate

    def should_sample(self, trace_id, operation_name=None):
        """在span创建时决策是否采样"""

        if self.strategy == "probabilistic":
            # 概率采样: 基于trace_id哈希
            trace_hash = int(trace_id[-16:], 16)
            return (trace_hash % 10000) < (self.rate * 10000)

        elif self.strategy == "rate_limiting":
            # 速率限制采样
            return self.rate_limiter.allow()

        elif self.strategy == "parent_based":
            # 跟随父span决策
            return self.parent_is_sampled()

    def create_span(self, name, context=None):
        # 创建span前立即决策
        trace_id = self.generate_trace_id()
        sampled = self.should_sample(trace_id, name)

        return Span(
            name=name,
            trace_id=trace_id,
            sampled=sampled,  # 决策立即生效
            parent=context
        )
```

### 2.2 概率采样配置

```yaml
# OpenTelemetry SDK配置
# otel-config-sdk.yaml

tracer_provider:
  sampler:
    # 基于概率的头部采样
    parent_based:
      root:
        trace_id_ratio_based:
          ratio: 0.1  # 10%采样率
      remote_parent_sampled: always_on
      remote_parent_not_sampled: always_off
      local_parent_sampled: always_on
      local_parent_not_sampled: always_off
```

### 2.3 优缺点分析

| 优点 | 缺点 |
|------|------|
| 性能开销极低 (<1%) | 无法捕获完整异常trace |
| 实现简单 | 异常可能在采样后被丢弃 |
| 资源消耗可预测 | 无法基于业务规则采样 |
| 适合大规模部署 | 采样决策缺乏context |

## 3. 尾部采样 (Tail Sampling)

### 3.1 Collector配置

```yaml
# tail-sampling-config.yaml
processors:
  tail_sampling:
    # 决策等待时间
    decision_wait: 10s

    # 同时追踪的最大trace数
    num_traces: 100000

    # 决策后trace保留时间
    expected_new_traces_per_sec: 1000

    policies:
      # 策略1: 采样所有错误
      - name: errors
        type: status_code
        status_code: {status_codes: [ERROR]}

      # 策略2: 采样慢请求 (>1s)
      - name: slow_requests
        type: latency
        latency: {threshold_ms: 1000}

      # 策略3: 特定业务采样 (10%)
      - name: important_services
        type: string_attribute
        string_attribute:
          key: service.name
          values: [payment, order]
          enabled_regex_matching: true

      # 策略4: 概率兜底
      - name: probabilistic
        type: probabilistic
        probabilistic: {sampling_percentage: 5.0}

      # 策略5: 组合策略 (错误或慢)
      - name: composite
        type: composite
        composite:
          sub_policy:
            - name: errors_policy
              type: status_code
              status_code: {status_codes: [ERROR]}
            - name: slow_policy
              type: latency
              latency: {threshold_ms: 500}
          operator: or
```

### 3.2 尾部采样决策流程

```
Trace进入 → 缓冲等待(10s) → 评估策略 → 决策
                                      ↓
                     ┌────────────────┴──────────────┐
                     ↓                               ↓
                  采样(Sampled)                 丢弃(Dropped)
                     ↓                               ↓
              导出到后端                      直接丢弃
                     ↓
              后续分析
```

### 3.3 内存优化策略

```go
// 尾部采样内存优化配置
// memory-efficient-config.yaml
processors:
  tail_sampling:
    decision_wait: 5s           # 减少等待时间
    num_traces: 50000           # 限制并发trace数

  # 前置: 头部采样减少负载
  probabilistic_sampler:
    sampling_percentage: 50.0   # 先50%头部采样

  # 后置: 尾部采样精筛
  tail_sampling:
    # 仅处理关键trace
    policies:
      - name: critical_only
        type: status_code
        status_code: {status_codes: [ERROR]}
```

## 4. 自适应采样 (Adaptive Sampling)

### 4.1 实现原理

```python
# 自适应采样算法
class AdaptiveSampler:
    """
    基于流量的自适应采样
    目标: 维持恒定采样率下的恒定QPS
    """

    def __init__(self, target_qps=1000, update_interval=60):
        self.target_qps = target_qps
        self.update_interval = update_interval
        self.current_rate = 0.1
        self.request_count = 0
        self.last_update = time.time()

    def should_sample(self, trace_id):
        # 基于当前采样率决策
        trace_hash = int(trace_id[-8:], 16)
        sampled = (trace_hash % 10000) < (self.current_rate * 10000)

        if sampled:
            self.request_count += 1

        # 定期调整采样率
        self._maybe_adjust_rate()
        return sampled

    def _maybe_adjust_rate(self):
        now = time.time()
        elapsed = now - self.last_update

        if elapsed >= self.update_interval:
            # 计算当前QPS
            current_qps = self.request_count / elapsed

            # 调整采样率
            if current_qps > self.target_qps * 1.1:
                # 流量过高, 降低采样率
                self.current_rate *= 0.8
            elif current_qps < self.target_qps * 0.9:
                # 流量过低, 提高采样率
                self.current_rate = min(1.0, self.current_rate * 1.2)

            # 记录指标
            print(f"采样率调整: {self.current_rate:.3f}, 当前QPS: {current_qps:.1f}")

            # 重置计数器
            self.request_count = 0
            self.last_update = now
```

### 4.2 按端点自适应采样

```python
# 按API端点自适应采样
class PerOperationSampler:
    """
    每个操作独立维护采样率
    适合API差异大的场景
    """

    def __init__(self):
        self.operation_samplers = {}
        self.default_target_qps = 10  # 每个端点10 qps

    def get_sampler(self, operation):
        if operation not in self.operation_samplers:
            # 新端点, 创建采样器
            self.operation_samplers[operation] = AdaptiveSampler(
                target_qps=self.default_target_qps
            )
        return self.operation_samplers[operation]

    def should_sample(self, trace_id, operation):
        sampler = self.get_sampler(operation)
        return sampler.should_sample(trace_id)

# 应用示例
sampler = PerOperationSampler()

# /api/health 高频端点 → 低采样率
# /api/payment 低频关键 → 高采样率
```

## 5. 策略选型决策树

### 5.1 选型流程图

```
开始选型
    │
    ▼
流量规模评估 ──→ 低流量(<1000 TPS) ──→ 全量采样
    │
    ▼
高流量(>1000 TPS)
    │
    ▼
资源预算评估 ──→ 严格限制 ──→ 头部采样 5%
    │
    ▼
中等预算
    │
    ▼
分析需求评估
    │
    ├── 需要异常分析 ──→ 尾部采样 (错误+慢请求)
    │
    ├── 需要业务分析 ──→ 按属性采样 + 尾部
    │
    └── 需要趋势分析 ──→ 自适应采样
    │
    ▼
最终推荐组合:
├── 头部采样 50% (减负)
├── 尾部采样 (错误100%, 慢10%)
└── 自适应兜底 5%
```

### 5.2 场景对照表

| 场景 | 推荐策略 | 配置建议 | 预期效果 |
|------|----------|----------|----------|
| 开发/测试 | 全量采样 | `always_on` | 100%覆盖 |
| 小规模生产 | 头部10% | `trace_id_ratio: 0.1` | 简单有效 |
| 中型微服务 | 头部+尾部 | 头部50% + 尾部关键 | 捕获异常 |
| 大型互联网 | 自适应+尾部 | 自适应+错误全采 | 动态优化 |
| 金融交易 | 业务采样 | 按交易金额/类型 | 关键业务 |
| IoT设备 | 边缘采样 | 设备端预采样 | 减少传输 |

## 6. 实战配置模板

### 6.1 推荐生产配置

```yaml
# production-sampling.yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317

processors:
  # 步骤1: 按服务预采样
  filter/health:
    spans:
      exclude:
        match_type: strict
        services: [health-check, metrics]

  # 步骤2: 头部采样减少负载
  probabilistic_sampler:
    sampling_percentage: 30.0

  # 步骤3: 尾部采样精筛
  tail_sampling:
    decision_wait: 5s
    num_traces: 50000
    policies:
      # 错误全采
      - name: errors
        type: status_code
        status_code: {status_codes: [ERROR]}

      # 慢请求采样
      - name: slow
        type: latency
        latency: {threshold_ms: 500}

      # 重要服务
      - name: critical
        type: string_attribute
        string_attribute:
          key: service.name
          values: [payment, auth, order]

      # 兜底概率
      - name: fallback
        type: probabilistic
        probabilistic: {sampling_percentage: 3.0}

exporters:
  otlp:
    endpoint: tempo:4317

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors:
        - filter/health
        - probabilistic_sampler
        - tail_sampling
      exporters: [otlp]
```

### 6.2 配置效果验证

```python
# 采样效果监控脚本
# validate_sampling.py

import requests
import json

def validate_sampling_effectiveness():
    """验证采样策略是否捕获关键trace"""

    # 查询采样后的trace
    query = '''
    {
      "traceQL": "{ status=error }"
    }
    '''

    response = requests.post(
        'http://tempo:3200/api/search',
        json=json.loads(query)
    )

    traces = response.json().get('traces', [])

    # 统计
    total_errors = len(traces)
    sampled_errors = sum(1 for t in traces if t.get('sampled'))

    # 计算覆盖率
    coverage = sampled_errors / total_errors * 100 if total_errors > 0 else 0

    print(f"错误Trace总数: {total_errors}")
    print(f"采样捕获数: {sampled_errors}")
    print(f"覆盖率: {coverage:.1f}%")

    # 告警阈值
    if coverage < 95:
        print("⚠️ 警告: 错误采样覆盖率不足95%")
        return False

    print("✅ 采样策略有效")
    return True

# 运行验证
validate_sampling_effectiveness()
```

## 7. 常见问题排查

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| Trace不完整 | 头部采样不一致 | 使用parent-based采样 |
| 内存溢出 | 尾部采样积压 | 减少decision_wait, 限制num_traces |
| 关键trace丢失 | 采样率过低 | 添加特定策略兜底 |
| 采样率波动 | 自适应参数不当 | 调整target_qps和interval |
